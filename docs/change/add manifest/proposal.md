# شناسنامه ماژول — یعنی module.json

## بخش ۱ — خلاصه‌ی پروژه‌ای که بر ما گذشت

داریم یه سیستم می‌سازیم که توش می‌شه ماژول‌های مختلف نصب کرد — مثل یه بازار کوچیک برای قطعه‌های نرم‌افزاری. یه ادمین می‌آد، یه ماژول رو آپلود می‌کنه، سیستم نصبش می‌کنه، و اون ماژول داخل داشبورد کار می‌کنه.

تا الان: سیستم نمی‌دونست با هر ماژول چیکار کنه. یه ZIP می‌اومد، سیستم حدس می‌زد — این فقط صفحه وبه، یا بک‌اند هم داره؟ هر بار یه دردسر جداگانه بود.


## بخش ۲ — ایده چی بود؟

**خلاصه:** گفتیم بذار هر ماژول یه «شناسنامه» کوچیک داشته باشه — یه فایل به اسم `module.json` — که توش بنویسه من کیم، چی دارم، چطور نصب بشم. سیستم اول این شناسنامه رو می‌خونه، بعد ادامه می‌ده.

**جونم برات بگه که:** مثلاً یه دولوپر یه ماژول گالری عکس می‌سازه. فقط یه صفحه وبه داره، هیچ بک‌اندی. وقتی آپلود کرد، سیستم شناسنامه رو ندید — پس یه شناسنامه ساده خودش ساخت و به دولوپر نشون داد: «ببین، یه شناسنامه برای ماژولت ساختم. می‌خوای تأیید کنی یا عوضش کنی?»

گالریست می‌گه «تأیید» — نصب انجام می‌شه.

حالا یه دولوپر دیگه یه ماژول پیش‌بینی آب‌وهوا می‌سازه که بک‌اند داره. شناسنامه‌اش می‌گه: «بک‌اند دارم، داخل Docker هستم, پورت ۳۰۰۰ رو می‌خوام.» سیستم می‌خونه, Docker رو راه می‌اندازه, همه چیز اتوماتیک می‌شه.

بدون شناسنامه? نصب متوقف می‌شه و راهنما نشون داده می‌شه. هیچ سوالی، هیچ wizardی، هیچ حدسی.


## بخش ۳ — چطور ممکنه?

**module.json برای همه ماژول‌هاست — نه فقط بک‌اندی‌ها**

اینپوینت مهم — هر ماژولی یه شناسنامه لازم داره، چه فقط صفحه وبه داشته باشه، چه بک‌اند هم داشته باشه.

**ماژول صرفاً صفحه وب (frontend-only)** — فقط فیلدهای اجباری خودش رو می‌خواهد (اسم، نسخه، فایل اصلی).
بقیه فیلدها خالی — سیستم خودش می‌فهمه بک‌اند نداره.

**ماژول با بک‌اند (frontend-backend)** — به فیلدهای اضافه هم نیاز داره: مسیر Docker، پورت، health check.
سازنده باید داک طراحی رو بخونه که Docker درونش به زبان ساده توضیح داده شده.

قدم به قدم چطور کار می‌کنه:

**قدم اول — آپلود ZIP:** سیستم چک می‌کنه `module.json` هست یا نه.
— اگه نبود: نصب متوقف — راهنما نشون داده می‌شه
— اگه بود ولی ناقص یا خراب بود: نصب متوقف — فیلد مشکل‌دار مشخص می‌شه
— اگه کامل بود: ادامه بده

**قدم دوم — نمایش و تأیید:** ادمین یه نمایش ساده می‌بینه: «این ماژول frontend-only هست، نسخه ۱.۲.۰، ساخته علیش.» می‌تونه تأیید کنه یا هر فیلدی رو عوض کنه.

**قدم سوم — نصب:** بعد تأیید ادمین، نصب انجام می‌شه. فرقی نمی‌کنه frontend باشه یا frontend-backend — سیستم روش صحیح رو خودش از شناسنامه می‌فهمه.


## بخش ۴ — خوبی‌ش چیه?

اقا چی بگم دیگه. ببین چی داریم:

- **سیستم دیگه حدس نمی‌زنه.** هر ماژول کارت شناسایی‌ش دستشه، سیستم می‌خونه، می‌فهمه، اجرا می‌کنه. تمام.
- **ادمین راحته.** قبل از نصب می‌بینه این ماژول چیه و چی می‌خواد — بدون اینکه با سازنده تماس بگیره. همینطور می‌تونه هر فیلدی رو مستقیم توی داشبورد ویرایش کنه.
- **سازنده هم راحته.** یه بار شناسنامه می‌نویسه، تا ابد ماژولش درست نصب می‌شه. دیگه نیازی نیست هر بار توضیح بده.
- **آپدیت راحته.** چون اطلاعات Docker و پورت توی شناسنامه‌ست، هر بار آپدیت هم درست اجرا می‌شه.


## بخش ۵ — چه پیدا کردم برات

مشابه دقیق پیدا نشد — ولی از اینا می‌شه الهام گرفت:

- **npm package.json** — همین کاری که npm انجام می‌ده! هر پکیج یه `package.json` داره که می‌گه چیه، چه نیازهایی داره، چطور اجرا بشه. ما داریم همین کار رو برای ماژول‌هامون می‌کنیم.
- **WordPress plugin header** — وردپرس توی ابتدای هر پلاگین یه بلوک توضیح داره که می‌گه این چیه. ما همین کار رو می‌کنیم ولی با فایل JSON جدا — که ماشین‌خوان‌تره.

کد آماده‌ای مستقیم پیدا نشد — منطقش ساده‌ست و از صفر نوشتنش کار زیادی نمی‌بره.


## بخش ۶ — ایده‌های جناب خودم یعنی AI

فک نکنم کسی دیگه‌ای بتونه اینقدر ایده بده. ماشالا به خودم:

- **ساخت خودکار شناسنامه برای frontend-only**: اگه سازنده شناسنامه نده یا ناقص بده، سیستم از روی ساختار ZIP یه شناسنامه پایه بسازه و برای تأیید بده. دیگه دیوار اول ورود نیست.
- **پیش‌نمایش شناسنامه قبل تأیید**: ادمین قبل از کلیک «نصب» یه خلاصه می‌بینه: «این ماژول frontend-only هست، پورت نمی‌خواد، کارت سفارشی داره.»
- **ویرایش شناسنامه توی داشبورد**: بعد نصب هم می‌شه برویم توی داشبورد و هر فیلدی رو تغییر بدیم. نیازی نیست دوباره ZIP بفرستیم.
- **نسخه‌بندی شناسنامه%**: هر بار ماژول آپدیت شد، نسخه قدیمی شناسنامه آرشیو بشه — بدونیم چی تغییر کرده.

(چی قرار نیست ساخته بشه: marketplace عمومی برای اشتراک‌گذاری ماژول‌ها، سیستم امتیازدهی یا هر چیزی شبیه plugin store — اینا برای بعداً.)


## بخش ۷ — سوالات و چالش‌ها

### ۱. راهنمای ZIP بدون شناسنامه چی باشه?

قراره که هر بار نصب متوقف بشه و راهنما نشون داده بشه — ولی محتوای راهنما چی باشه?

- الف — «فایل شناسنامه (module.json) وجود ندارد. برای ساختن آن داک طراحی را بخوانید … [لینک]»
- ب — «می‌توانید کد ماژول را به AI بدهید تا شناسنامه را بسازد … [لینک]»
- **نظر شما چیه?**

### ۲. اگه سازنده بک‌اند داره ولی نمی‌دونه Docker چیه?

اقا مثل اینکه شما نمی‌خوای کار کنی داری بهونه می‌آری! ولی جدی جدی:

- الف — داک طراحی یه بخش جداگانه داره که Docker رو به زبان ساده توضیح می‌ده — سازنده باید اون رو بخونه
- ب — اگه Docker نداره، نصب متوقف بشه و پیام راهنما نشون داده بشه
- **نظر شما چیه?**

### ۳. ادمین بعد از نصب چقدر می‌تونه شناسنامه رو تغییر بده?

- الف — فقط فیلدهای کم‌خطر (مثل اسم نمایشی) قابل تغییر باشه
- ب — هر فیلدی قابل تغییر باشه — با re-validate بعد از ذخیره
- **نظر شما چیه?**
