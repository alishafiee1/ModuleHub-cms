# طراحی شناسنامه ماژول — `module.json`

> نسخه: ۱.۱ — تاریخ: ۱۴۰۴/۰۳/۲۹ — نویسنده: تیم RODI


## بخش ۰ — پروژه در یه نگاه

ModuleHub یه سیستم مدیریت محتواست که توش می‌شه ماژول‌های مستقل نصب کرد و به هم وصل کرد. یه ادمین یه ZIP آپلود می‌کنه، سیستم نصب می‌کنه، و ماژول داخل داشبورد کار می‌کنه. الان در مرحله MVP هستیم و این فایل مربوط به طراحی سیستم شناسنامه ماژول‌هاست.


## بخش ۱ — ماجرا از چه قراره؟

یادته تو پروپوزال گفتیم هر ماژول باید یه «شناسنامه» داشته باشه؟ خب اینجا قراره بگیم این کار دقیقاً با چه ابزارها، چه منطقی و چه رفتاری انجام می‌شه.

تو این فایل می‌خوای دید که: فرمت دقیق `module.json` چیه، سیستم موقع آپلود چطور کار می‌کنه، و ادمین چطور با شناسنامه تعامل می‌کنه.


## بخش ۲ — طرح چطوری کار می‌کنه؟

### فرمت شناسنامه (module.json)

ببین، شناسنامه یه فایل JSON ساده‌ست که داخل ZIP ماژول قرار می‌گیره. دو نوع داریم:

**نوع اول — frontend-only**: فقط صفحه وب، بدون بک‌اند. فیلدهای اجباری:

```json
{
  "name": "gallery-v2",
  "version": "1.0.0",
  "type": "frontend-only",
  "entry": "index.html"
}
```

**نوع دوم — frontend-backend**: صفحه وب + بک‌اند داخل Docker. علاوه بر فیلدهای بالا:

```json
{
  "name": "weather-forecast",
  "version": "1.0.0",
  "type": "frontend-backend",
  "entry": "index.html",
  "backend": {
    "dockerfile": "backend/Dockerfile",
    "ports": ,
    "healthCheck": "/health"
  }
}
```

راستش: فیلد `ports` یه **آرایه** از اعداده — ماژول می‌تونه چند پورت نیاز داشته باشه (مثلاً HTTP روی ۳۰۰۰ و WebSocket روی ۴۰۰۰). اگه `backend` نبود، سیستم خودش می‌فهمه که frontend-only هست.


### موتور بررسی شناسنامه و چک پورت

یه سرویس ساده که موقع آپلود ZIP این کارها رو به ترتیب انجام می‌ده:

1. داخل ZIP دنبال `module.json` می‌گرد
2. اگه پیدا نشد: در مورد frontend-only یه نمونه پایه خودش می‌سازه و برای تأیید می‌فرسته؛ در مورد frontend-backend نصب متوقف می‌شه و راهنما نشون داده می‌شه.
3. اگه خطای اعتبارسنجی داشت: فیلد مشکل‌دار مشخص می‌شه — نصب متوقف.
4. **چک آزاد بودن پورت‌ها:** قبل از راه‌اندازی Docker، سیستم همه پورت‌های داخل `ports` رو با تلاش برای bind کردن یه socket موقت چک می‌کنه. اگه یه پورت اشغال بود: نصب همینجا متوقف می‌شه (Docker اجرا نمی‌شه) و پیام: «پورت X در حال استفاده توسط برنامه دیگری است. لطفاً مقدار دیگری در شناسنامه مشخص کنید» نمایش داده می‌شود.
5. اگه کامل بود: خلاصه شناسنامه رو به ادمین نشون می‌ده و منتظر تأیید می‌ماند.


### نمایش و ویرایش شناسنامه در داشبورد

بعد از نصب، ادمین می‌تونه بره صفحه ماژول بره و شناسنامه رو ببینه. هر فیلدی قابل ویرایش هست — بعد ذخیره سیستم خودش re-validate می‌کنه. نسخه قدیمی شناسنامه هم آرشیو می‌شه.

وابستگی: این بخش به انجن بررسی وابسته‌ست — اگه انجن در دسترس نبود، ویرایش ذخیره می‌شه ولی re-validate عقب انجام نمی‌شه.


### سیستم نصب و فایروال

بعد از تأیید ادمین، سیستم بر اساس فیلد `type` تصمیم می‌گیرد:

- **frontend-only**: فایل‌ها رو استخراج می‌کنه و در مسیر استاتیک سرویس می‌ده.
- **frontend-backend**: سرویس Docker رو راه می‌اندازد، health check انجام می‌ده، بعد تأیید نصب نهایی انجام می‌شه.

**باز کردن پورت روی فایروال:**
بعد از اینکه سرویس Docker بالا آمد و health check پاس شد، سیستم باید پورت‌های داخل `ports` رو روی فایروال سرور باز کنه:

- **Linux (ufw)**: اجرای `ufw allow [PORT]/tcp` به ازای هر پورت
- **Windows Server**: اجرای دستور `netsh advfirewall` یا PowerShell

نکته مهم: این دستورات نیاز به دسترسی root/admin دارن — سرویس نصب باید با این سطح دسترسی اجرا بشه. موقع حذف ماژول هم همین پورت‌ها باید از فایروال **بسته** بشن.


## بخش ۳ — با چی می‌سازیمش؟

- **فرمت**: JSON خالص. چرا YAML نه؟ چون پایتون/Node.js/Go همه نیتیو JSON رو می‌فهمن، ولی YAML نیاز با لایبرری اضافه داره.
- **اعتبارسنجی**: JSON Schema.
- **اجرای کانتینر**: Docker SDK.


## بخش ۴ — جواب سوال‌های پروپوزال

- **اگه ZIP شناسنامه نداشت چی؟** اگه بک‌اند نداشته باشه، خودمون یکی با اسم پوشه می‌سازیم.
- **اگه سازنده Docker نمی‌دونست؟** تو فاز اول، ماژول بک‌انددار بدون Dockerfile قبول نمی‌شه.
- **ادمین چقدر می‌تونه ویرایش کنه؟** همه فیلدها بازن، ولی اگه `ports` رو عوض کنه، کانتینر باید ری‌استارت بشه و پورت‌های فایروال هم آپدیت بشن.


## بخش ۵ — خارج از اسکوپ (Out of scope)

- مارکت‌پلیس آنلاین برای دانلود مستقیم
- سیستم امتیازدهی و نظرات
- احراز هویت اختصاصی در صفحه نصب


## بخش ۶ — ریسک‌ها و نکات مراقبتی

| دسته | سناریو | رفتار سیستم |
|------|--------|-------------|
| [ورودی] | فایل `module.json` خراب | نصب لغو، پیام: "فایل شناسنامه نامعتبر است." |
| [ورودی] | فیلد `ports` آرایه نبود یا خالی بود | پیام «فیلد ports باید آرایه‌ای از اعداد باشد» — نصب متوقف |
| [سرویس] | یکی از پورت‌ها قبل از نصب اشغال بود | پیام «پورت [X] در حال استفاده است» — نصب متوقف، Docker اجرا نمی‌شه |
| [سرویس] | دسترسی به تنظیم فایروال رد شد | پیام «اجازه تغییر فایروال نبود» — نصب لغو یا هشدار برای تنظیم دستی |
| [داده] | پورت تغییر کرد ولی کانتینر آپدیت نشد | هنگام ویرایش شناسنامه، تغییر پورت کانتینر رو Force Restart می‌کنه. |


## بخش ۷ — تست‌های تأییدیه (Acceptance Tests)

1. **تست آپلود ZIP بدون شناسنامه:** سیستم باید فایل رو قبول کنه، شناسنامه دیفالت بسازه و تو داشبورد نشون بده.
2. **تست شناسنامه خراب:** سیستم باید با خطای مشخص جلوی نصب رو بگیره.
3. **تست تداخل پورت:** آپلود ماژول با پورت 3000 وقتی اشغاله -> سیستم باید قبل از اجرای داکر خطا بده و نصب را لغو کنه.
4. **تست فایروال:** نصب ماژول با پورت 4500 -> بررسی باز شدن پورت تو ufw/Windows Firewall.
5. **تست ویرایش:** تغییر نام و پورت ماژول در داشبورد -> فایل شناسنامه باید آپدیت بشه، کانتینر ری‌استارت بشه و فایروال پورت جدید رو اعمال کنه.
