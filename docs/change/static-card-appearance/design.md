# طراحی — ظاهر سفارشی ثابت کارت

> **مسیر:** `docs/change/static-card-appearance/design.md`  
> **یک خط:** توضیح می‌دهد ادمین چطور قالب بصری ثابت می‌نویسد، سرور چطور آن را پاک‌سازی و ذخیره می‌کند، و مرورگر چطور بدون اجرای دستور زنده روی بوم نشان می‌دهد.  
> **پیش‌نیاز:** [proposal.md](./proposal.md)  
> **رفتار دقیق روی صفحه** (رنگ دکمه، اندازه ویرایشگر، متن toast) → بعداً در `ui-behavior.md` / `docs/UI-behavior.md`  
> **جدا:** مارک‌داون → [folder-card-management](../1405-03-24-folder-card-management/design.md) · کارت پویا → [card-live-customization](../card-live-customization/design.md)

---

فرض کن ادمین روی کارت ماژول «دماسنج» چرخ‌دنده زده و «ویرایش ظاهر کارت» را باز کرده. در بالای فرم می‌بیند روش نمایش روی «پیش‌فرض» است؛ آن را به «ظاهر سفارشی» می‌برد. فیلد مارک‌داون محو می‌شود و به‌جایش یک جعبهٔ متن برای قالب و کنارش پیش‌نمایش کوچک همان کارت روی بوم می‌آید. ادمین می‌نویسد عنوان بزرگ پایین کارت بیفتد و پس‌زمینه از «تصویر کارت» پر شود — نه با آدرس اینترنتی، بلکه با همان کلمهٔ ثابتی که سیستم به فایل آپلودشده وصل می‌کند. دکمهٔ ذخیره را می‌زند؛ سرور قالب را بررسی می‌کند، خطرناک‌ها را حذف می‌کند، در فایل چیدمان سایت ذخیره می‌کند. بازدیدکننده بعد از رفرش همان کارت را می‌بیند؛ هیچ دستوری داخل کارت اجرا نمی‌شود.

این سند همان مسیر را از کلیک ادمین تا ذخیره و نمایش توضیح می‌دهد.

---

## محدوده

**داخل این طراحی:** انتخاب روش نمایش per کارت، ویرایش و پیش‌نمایش قالب توسط ادمین اصلی، پاک‌سازی و ذخیره روی node چیدمان، رندر امن روی بوم برای پوشه و ماژول.

**خارج از این طراحی:** رنگ و اندازهٔ دقیق دیالوگ (ui-behavior)، چرا محصول لازم است (proposal)، دادهٔ لحظه‌ای از ماژول (card-live-customization)، قالب HTML داخل ZIP ماژول (archive معلق).

---

## تصویر کلی

```
ادمین قالب را می‌نویسد و ذخیره می‌کند
        ↓
سرور: پاک‌سازی + سقف حجم + ذخیره در چیدمان سایت
        ↓
مرورگر: جای‌نگهدارها → آدرس امن + رندر داخل محفظهٔ ایزوله
        ↓
بازدیدکننده: همان کارت روی بوم — کلیک مثل قبل
```

هر کارت در هر لحظه **یکی** از دو حالت را دارد:

| حالت | روی بوم چه می‌بینیم |
|------|---------------------|
| **پیش‌فرض** | آیکون، عنوان، توضیح مارک‌داون (یا پس‌زمینهٔ پالت — مستقل از این change) |
| **ظاهر سفارشی** | فقط خروجی قالب ادمین؛ مارک‌داون توضیح کارت نمایش داده نمی‌شود |

چرخ‌دنده، نشانگر وضعیت ماژول، و drag/resize گرید مثل سیستم فعلی روی لایهٔ بیرونی کارت می‌مانند — قالب فقط **بدنهٔ داخلی** کارت را پر می‌کند.

---

## داده‌ای که ذخیره می‌شود

روی همان nodeای که الان `cardDescription` و `cardBackground` دارد، فیلدهای جدید اضافه می‌شود — مفهومی:

- **روش نمایش:** `default` یا `custom-static`
- **متن قالب:** رشتهٔ HTML/CSS (با سقف حجم — پیشنهاد حدود ۱۶–۳۲ کیلوبایت)
- **نسخهٔ قالب** (اختیاری): برای migrate آینده اگر قواعد عوض شد

اگر ادمین از ظاهر سفارشی به پیش‌فرض برگردد، متن قالب می‌تواند در دیتابیس/فایل بماند ولی **نادیده** گرفته شود — تا اگر دوباره سوییچ کرد، کارش از بین نرود.

---

## جای‌نگهدارهای مجاز (عکس و متن)

به‌جای آدرس آزاد، ادمین فقط **توکن‌های ثابت** در قالب می‌گذارد. سرور هنگام رندر آن‌ها را به مقدار واقعی تبدیل می‌کند:

| توکن | منبع |
|------|------|
| `{{title}}` | نام کارت از چیدمان |
| `{{description}}` | متن توضیح ذخیره‌شده (برای استفادهٔ اختیاری داخل قالب؛ در حالت custom-static نمایش خودکار مارک‌داون نداریم) |
| `{{thumbnail}}` | تصویر کارت ماژول اگر آپلود شده؛ وگرنه خالی یا آیکون پیش‌فرض |
| `{{cardBackground}}` | همان `cardBackground` فعلی (رنگ یا عکس آپلودی از پالت) |
| `{{folderIcon}}` / `{{moduleIcon}}` | آیکون نوع کارت |
| `{{statusBadge}}` | فقط ماژول — HTML امن از سیستم (نه از دست ادمین) |

هر `img` یا `url()` در CSS که توکن نباشد و به مسیر خارج از CMS اشاره کند، هنگام ذخیره **رد** یا **حذف** می‌شود. `data:` برای عکس در فاز اول مجاز نیست — حجم و سوءاستفاده.

---

## فرایند ویرایش توسط ادمین

۱. فقط **ادمین اصلی** وارد شده می‌تواند قالب را ببیند و عوض کند — همان سطح دسترسی ویرایش نام/توضیح پوشه.

۲. در دیالوگ، سوییچ روش نمایش → اگر `custom-static`، ویرایشگر قالب + پیش‌نمایش فعال می‌شود.

۳. پیش‌نمایش همان مسیر رندر بازدیدکننده را طی می‌کند (پاک‌سازی + جای‌نگهدار) — نه «اعتماد به مرورگر ادمین».

۴. ذخیره → درخواست به سرور با توکن امنیتی همان فرم‌های ادمین.

۵. سرور: اعتبارسنجی طول، پاک‌سازی، جایگزینی توکن‌های ناشناس، ذخیرهٔ atomic در فایل چیدمان.

۶. پاسخ موفق → toast → کارت روی بوم بدون رفرش کامل صفحه به‌روز می‌شود (همان الگوی PATCH کارت‌های فعلی).

اگر پاک‌سازی همه‌چیز را خالی کرد یا ساختار شکست، ذخیره **رد** می‌شود و پیام خطای قابل فهم می‌آید — نه ذخیرهٔ خام.

---

## پاک‌سازی و «بدون دستور زنده»

قالب **فقط ظاهر** است. سرور و مرورگر هر دو نقش دارند:

**هنگام ذخیره (سرور):** حذف تگ‌های `script`، `iframe`، `object`، `embed`؛ حذف handlerهای رویداد (`onclick` و مشابه)؛ حذف `javascript:` در لینک؛ محدود کردن تگ‌ها به لیست سفید (مثلاً `div`, `span`, `p`, `h1`–`h6`, `img`, `a`, `ul`, `ol`, `li`, `strong`, `em`, `br`, `style` داخل محدوده)؛ CSS فقط در `style` تگ یا بلوک `<style>` با ممنوعیت `@import` و `url()` به منبع خارجی.

**هنگام نمایش (مرورگر):** رندر داخل **محفظهٔ ایزوله** — Shadow DOM یا div با scope — بدون `innerHTML` ناامن روی کل صفحه؛ **بدون** iframe با اجازهٔ اسکریپت (آن مخصوص کارت پویاست). هیچ اسکریپتی از قالب اجرا نمی‌شود حتی اگر ادمین دور زدن امتحان کند.

این ترکیب یعنی حتی اگر کسی قالب مخرب بفرستد، بازدیدکننده کوکی ادمین یا صفحهٔ اصلی را در خطر نمی‌اندازد.

---

## رندر روی بوم

وقتی کارت با `custom-static` ساخته می‌شود:

- لایهٔ بیرونی (گرید، چرخ‌دنده، دستگیرهٔ ویرایش) مثل کارت عادی.
- داخل `.card-content` به‌جای آیکون+عنوان+مارک‌داون، یک ظرف «قالب سفارشی» قرار می‌گیرد.
- قالب از سرور یا از دادهٔ layout با جای‌نگهدارهای resolve‌شده پر می‌شود.
- اگر `overflow` لازم است، داخل همان سلول گرید با `overflow: hidden` — تا گرید نشکند.
- حالت تاریک سایت: اگر ادمین در قالب رنگ ثابت روشن نوشته، سیستم فعلاً دست نمی‌زند — در ui-behavior می‌توان هشدار پیش‌نمایش دارک داد.

کلیک روی بدنهٔ کارت: مثل قبل — پوشه navigate، ماژول هدایت. کلیک روی چرخ‌دنده: `stopPropagation` مثل الان.

---

## درخواست‌های سرور (فقط ادمین)

| عمل | معنی |
|-----|------|
| خواندن قالب فعلی | برای پر کردن ویرایشگر |
| ذخیرهٔ قالب + روش نمایش | پاک‌سازی، validate، PATCH روی node |
| پیش‌نمایش بدون ذخیره (اختیاری) | همان رندر نهایی روی متن موقت — برای دیالوگ |

همهٔ این‌ها فقط با session ادمین اصلی. بازدیدکننده قالب خام را هرگز از API جدا نمی‌گیرد — فقط HTML resolve‌شده داخل صفحهٔ اصلی.

---

## وقتی خطا می‌خورد

| وضعیت | برای کاربر یعنی چه |
|-------|---------------------|
| قالب خالی بعد از پاک‌سازی | ذخیره رد؛ پیام «قالب معتبر نیست» |
| قالب ذخیره شده ولی بعداً خراب شد (دستی در فایل) | روی بوم fallback به پیش‌فرض + لاگ سرور |
| تصویر توکن وجود ندارد | جای خالی یا placeholder خاکستری — نه 404 شکسته |
| ادمین session ندارد | ریدایرکت لاگین مثل بقیهٔ عملیات ادمین |

---

## ریسک‌های فنی

اگر دو PATCH هم‌زمان روی یک node برسد، ممکن است یکی دیگری را بپوشاند — همان ریسک فعلی `site-layout.json`؛ راه‌حل: خواندن نسخهٔ تازه قبل از write یا پیام «دوباره تلاش کن».

قالب خیلی بزرگ می‌تواند فایل layout را سنگین کند — سقف حجم و هشدار در ویرایشگر لازم است.

CSS پیچیده با `position: absolute` ممکن است روی گرید ۳×۳ کوچک بریزد — پیش‌نمایش با ابعاد واقعی سلول کارت در هر breakpoint کمک می‌کند.

| اسکریپت | هرگز | داخل iframe sandbox | خیر |

## سناریوهای آزمون (مسیر سیستم)

۱. ادمین: سوییچ به custom-static → قالب با `{{title}}` و `{{thumbnail}}` → ذخیره → بوم و بازدید مهمان یکسان.

۲. قالب حاوی `script` → ذخیره رد یا script حذف و اگر خالی شد رد.

۳. سوییچ برگشت به default → مارک‌داون دوباره دیده می‌شود؛ قالب در layout مانده ولی استفاده نمی‌شود.

۴. بیست کارت روی صفحه، سه تا custom — بارگذاری اولیه بدون timeout اضافه نسبت به baseline.
