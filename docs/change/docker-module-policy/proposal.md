# سیاست Docker-Only — هر بک‌اند داخل جعبه‌ی امن خودش

***

## این تغییر چیه؟ — داستان از کجا شروع شد

الان اگه یه ماژول Node.js یا Python بخواد روی سرور اجرا بشه، CMS مستقیم دستور node یا python رو روی سرور اجرا می‌کنه. این ماژول به همه منابع سرور دسترسی داره — فایل‌ها، شبکه، CPU، RAM — بدون هیچ محدودیتی.

یه ماژول بد (یا حتی یه ماژول خوب با باگ) می‌تونه:
- کل CPU سرور رو بخوره و بقیه سرویس‌ها رو کُند کنه
- به فایل‌های سیستمی دسترسی داشته باشه
- به شبکه داخلی وصل بشه
- سرویس‌های دیگه رو مختل کنه

Docker راه‌حله. هر ماژول داخل یه «جعبه» جداگانه اجرا میشه. منابعش محدوده، دسترسی‌هاش کنترل‌شده، اگه کرش کنه بقیه سیستم آسیب نمی‌بینه.

***

## چه مشکلی رو حل می‌کنه؟

قبل از این تغییر:
- ماژول Node.js مستقیم روی سرور اجرا میشه
- هیچ محدودیت CPU/RAM نیست
- یه ماژول معیوب می‌تونه کل سرور رو مختل کنه
- توی داک‌های اصلی تناقض بین «اجرای مستقیم» و «Docker» وجود داره

بعد از این تغییر:
- هر ماژول با بک‌اند حتماً داخل Docker اجرا میشه
- محدودیت CPU و RAM از طریق پنل ادمین قابل تنظیمه
- ماژول بدون Docker نصب نمیشه و راهنما نشون داده میشه
- ادمین تنظیمات Docker هر ماژول رو از پنل می‌بینه و تأیید می‌کنه

***

## قانون Docker-Only — دقیقاً چی میشه

### تعریف دقیق

هر ماژولی که hasBackend: true داره الزاماً باید dockerComposePath معتبر در module.json داشته باشه.

ماژول‌های frontend-only که فقط HTML/CSS/JS استاتیک دارن از این قانون مستثنا هستن. اونا اصلاً پروسه‌ای روی سرور ندارن.

### جریان تصمیم‌گیری در آپلود

ZIP رسید ←
آیا module.json داره؟
  نه ← خطا: «شناسنامه ماژول پیدا نشد» + لینک راهنما
  بله ←
آیا hasBackend: true داره؟
  نه ← نصب ادامه پیدا می‌کنه (frontend-only)
  بله ←
آیا dockerComposePath معتبر داره؟
  بله ← نصب ادامه پیدا می‌کنه
  نه ← توقف + نمایش پیام راهنما به کاربر

## پیام راهنما به کاربر

وقتی ماژول بک‌اند داره ولی Docker نداره، این پیام نشون داده میشه:

***

⚠️ این ماژول کد بک‌اند دارد

کدی که ارسال کرده‌اید دارای کدهای بک‌اند است که بر روی سرور نیاز به اجرا و مدیریت دارند.
برای نصب امن، ابتدا باید آن را در یک بسته Docker با محدودیت‌های لازم قرار دهید.

می‌توانید:
- راهنمای داکرایز کردن را مطالعه کنید
- یا این راهنما را به یک agent هوش مصنوعی بدهید تا این کار را برایتان انجام دهد

پس از آماده‌سازی، فایل ZIP جدید را دوباره آپلود کنید.

## تنظیمات Docker در پنل ادمین

بعد از نصب موفق، ادمین از طریق دکمه ⚙ روی کارت ماژول می‌تونه این موارد رو ببینه و تغییر بده:

وضعیت container:
- روشن / خاموش / ارور

محدودیت‌های منابع:
- حداکثر CPU — مثلاً 0.5 یعنی نصف یه هسته
- حداکثر RAM — مثلاً 256m

شبکه:
- پورت‌هایی که container در معرض دید قرار میده

متغیرهای محیطی (Environment Variables):
- نمایش کلیدها و مقادیر
- مقادیر حساس مثل رمزها با *** پوشیده میشن
- ادمین می‌تونه تأیید یا اصلاح کنه قبل از اجرا

### نمونه تنظیمات در module.json

    "backend": {
      "dockerComposePath": "backend/docker-compose.yml",
      "internalPort": 3000,
      "healthCheck": "/health",
      "resources": {
        "cpuLimit": "0.5",
        "memLimit": "256m"
      }
    }

## جایگزینی Start/Stop با Docker

دستورهای فعلی که باید حذف بشن:
- systemd-run node index.js
- systemd-run python app.py
- هر اجرای مستقیم Node/Python/PHP

دستورهای جدید:
- Start: docker compose -f [مسیر] up -d
- Stop: docker compose -f [مسیر] down
- Restart: docker compose -f [مسیر] restart
- Logs: docker logs [container-name]

ماژول‌های frontend-only دست نخورن — اونا اصلاً دستور اجرایی ندارن.

## معیار موفقیت

- ZIP با بک‌اند ولی بدون Docker: توقف نصب + پیام راهنما نشون داده میشه
- ZIP با Docker درست: نصب موفق، container بالا میاد
- دکمه Start: docker compose up اجرا میشه نه node مستقیم
- دکمه Stop: docker compose down اجرا میشه
- لاگ از docker logs خونده میشه نه از فایل مستقیم
- ادمین CPU و RAM رو از پنل می‌بینه و می‌تونه محدود کنه
- ماژول‌های frontend-only بدون تغییر کار می‌کنن
