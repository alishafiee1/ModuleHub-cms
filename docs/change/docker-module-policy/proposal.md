# سیاست Docker-Only — هر بک‌اند داخل جعبه‌ی امن خودش

***

## بخش ۱ — قبلاً اینجا بودیم

ModuleHub CMS یه سیستمه که روی سرور نصب میشه و بهت اجازه میده ماژول‌های مختلف رو آپلود کنی، نصب کنی، اجرا کنی، و از یه داشبورد مدیریتشون کنی. فکرش رو بکن مثل یه «فروشگاه اپ» ولی برای سرور خودت — هر ماژول یه کارت روی صفحه داره که می‌تونی باهاش کار کنی.

تا الان چیزای زیادی ساختیم: آپلود ماژول (M2)، اجرا و توقف (M3)، کش پکیج‌ها (M4)، پنل مدیریت (M5)، بکاپ (M6)، احراز هویت (M14)... یه سیستم نسبتاً کامل.

**تا الان:** وقتی یه ماژول Node.js یا Python آپلود میشد، CMS مستقیم دستور `node index.js` یا `python app.py` رو روی سرور اجرا می‌کرد. بدون محدودیت. بدون دیوار. ماژول هر کاری دلش می‌خواست می‌تونست بکنه.

***

## بخش ۲ — یه روز یه ماژول دیوونه شد

### الف) ایده از کجا اومد؟

از یه جای ترسناک: واقعیت. وقتی ساختار `module.json` (M16) داشت طراحی می‌شد، یه سوال اساسی مطرح شد — «اگه یه ماژول بد اومد چی؟» جواب ساده بود: «فعلاً هیچی نمی‌تونیم بکنیم.» این ایده اومد که باید یه دیوار بکشیم بین ماژول‌ها و بقیه سیستم.

### ب) جونم برات بگه که:

تصور کن یه دولوپر یه ماژول می‌سازه که قراره داده‌های دما رو از یه سنسور IoT بگیره و نشونشون بده. کد خوبیه. ولی یه باگ داره — یه حلقه‌ی بی‌نهایت که توی یه شرایط خاص فعال میشه. 

الان چی میشه؟ ماژول CPU سرور رو می‌بلعه. پنل مدیریت کُند میشه. بقیه ماژول‌ها که درست کار می‌کردن، یواش یواش جواب نمیدن. کاربرها می‌پرسن «مشکل از کجاست؟» و تو داری دنبال یه باگ توی یه ماژول از بین ده‌تا می‌گردی.

با Docker؟ اون ماژول داخل جعبه‌ی خودشه. CPU‌اش محدوده. RAM‌اش سقف داره. اگه دیوونه بشه، توی جعبه می‌مونه. بقیه سرویس‌ها؟ حتی نمی‌فهمن که یه همسایه دیوونه دارن.

***

## بخش ۳ — دیوار چطور کشیده میشه؟

منطق کار اینه: وقتی یه ZIP آپلود میشه، CMS اول `module.json` رو چک می‌کنه (این از M16 میاد). اگه ماژول بک‌اند داشت (`hasBackend: true`)، باید حتماً یه `dockerComposePath` معتبر داشته باشه.

اگه نداشت؟ نصب متوقف میشه و یه پیام راهنما نشون داده میشه که توضیح میده چطور ماژول رو «داکرایز» کنی — یا اینکه این راهنما رو به یه AI agent بدی تا خودش کارو بکنه.

اگه داشت؟ نصب ادامه میده. دکمه Start دیگه `node index.js` اجرا نمی‌کنه — بجاش `docker compose up -d` رو صدا می‌زنه. Stop هم `docker compose down`. لاگ‌ها از `docker logs` میان نه از فایل مستقیم.

ادمین هم از طریق پنل ⚙ هر ماژول می‌تونه محدودیت CPU و RAM رو ببینه و تغییر بده — بدون اینکه دست به سرور بزنه.

ماژول‌های `frontend-only`؟ اونا اصلاً بک‌اند ندارن. هیچ تغییری برای اونا نیست.

***

## بخش ۴ — چرا این ایده شیرینه

اقا چی بگم. وقتی یه سیستم رو می‌سازی که چند نفر باهاش کار می‌کنن، دیگه نمی‌تونی به «کد خوب همه» اعتماد کنی. یه ماژول بد کافیه که کل چیزی که ساختی رو بپوشونه.

خوبی‌های این تغییر:

- **امنیت واقعی** — ماژول نمی‌تونه به فایل‌های سیستمی یا سرویس‌های دیگه دسترسی داشته باشه
- **کنترل منابع** — ادمین از پنل CPU و RAM هر ماژول رو تنظیم می‌کنه
- **پیش‌بینی‌پذیری** — همه ماژول‌ها یه جور یکسان اجرا میشن، نه هر کدوم یه جور
- **راحتی دولوپر** — سازنده ماژول Docker می‌نویسه، مطمئنه همه‌جا یکسان کار می‌کنه
- **راهنمای AI** — پیام خطا طوری طراحی شده که بشه داد به AI agent تا Dockerfile بسازه

***

## بخش ۵ — ببینیم دیگران چطور حل کردن

پروژه‌هایی که دقیقاً همین مشکل رو حل کردن:

- **[Coolify](https://coolify.io/)** — یه self-hosted platform که هر اپ رو داخل Docker deploy می‌کنه. ساختارشون برای «هر سرویس = یه container» خیلی شبیه چیزیه که ما می‌خوایم
- **[Portainer](https://www.portainer.io/)** — مدیریت Docker از طریق UI. ایده‌ی نمایش CPU/RAM و وضعیت container از اینجا می‌تونه الهام بگیره
- **[CapRover](https://caprover.com/)** — deploy ماژول‌ها با Docker Compose از طریق dashboard. خیلی به معماری ما نزدیکه

کد آماده‌ای که میشه ازش استفاده کرد: کتابخونه‌ی `dockerode` برای Node.js — بجای اجرای دستور shell، مستقیم با Docker API حرف می‌زنه. تمیزتره.

***

## بخش ۶ — ایده‌های جناب خودم یعنی AI

فک نکنم کسی اینقدر ایده می‌داد. ماشالا به خودم:

- **قفل کردن network داخلی** — Docker network هر ماژول جدا باشه. ماژول A نتونه به database ماژول B وصل بشه مگه اینکه صراحتاً اجازه داده بشه
- **Health check خودکار** — وقتی ماژول `healthCheck: /health` داره توی `module.json`، CMS هر چند دقیقه یه بار چک کنه زنده‌ست یا نه. کارت ماژول رنگ قرمز بشه اگه جواب نداد
- **Auto-restart policy** — اگه container کرش کرد، Docker خودش restart کنه (`restart: unless-stopped`)
- **پیام راهنما با مثال Dockerfile** — وقتی پیام خطا نشون داده میشه، یه `Dockerfile` و `docker-compose.yml` نمونه هم inline نشون داده بشه — نه فقط لینک، بلکه کد آماده کپی کن
- **Resource dashboard کلی** — یه صفحه که بگه «الان مجموع CPU و RAM همه ماژول‌ها چقدره» — مثل یه نگاه از بالا به کل سیستم

(چی قرار نیست ساخته بشه: ساختن Dockerfile به صورت خودکار توی CMS، مدیریت Docker registry، و image building — اینا خارج از اسکوپ این پروپزاله.)

## بخش ۷ — سوالات و چالش‌ها که نمیشه ازشون فرار کرد

### ۱. ماژول‌های قدیمی که الان بدون Docker دارن کار می‌کنن چی میشن؟

- الف — هشدار میدیم ولی فعلاً اجازه میدیم کار کنن (migration تدریجی)
- ب — از روز اول قانون رو اجباری می‌کنیم، هر کی Docker نداره باید از نو آپلود کنه
- پ — یه deadline مشخص میذاریم (مثلاً بعد از ۳۰ روز قفل میشن)
- نظر شما چیه؟

### ۲. اگه Docker روی سرور نصب نبود چی؟

- الف — CMS اول چک کنه Docker هست یا نه، اگه نبود پیام راهنما نصب بده
- ب — این رو توی prerequisites اضافه کنیم و از اول مستند باشه
- پ — نظر شما چیه؟

### ۳. ادمین می‌خواد CPU رو از پنل تغییر بده، ولی container داره اجرا میشه

- الف — تغییر فوری اعمال بشه (docker update) بدون restart
- ب — پیام بده «برای اعمال تغییر، Stop و Start کن»
- پ — نظر شما چیه؟

### ۴. اگه `docker-compose.yml` ماژول پورت مهمی مثل ۸۰ یا ۴۴۳ رو expose کرد؟

- الف — CMS پورت‌های حساس رو بلاک کنه
- ب — هشدار بده ولی اجازه بده
- پ — Super Admin بتونه لیست پورت‌های ممنوع رو تعریف کنه
- نظر شما چیه؟
