# طراحی فنی فرآیند نصب و اجرای ماژول غیر داکری

## هدف

این سند طراحی فنی فرآیند نصب، ثبت، اجرا و به‌روزرسانی ماژول‌های غیر داکری را مشخص می‌کند.  
منظور از ماژول غیر داکری، ماژولی است که به‌صورت یک برنامه عادی روی سیستم اجرا می‌شود و برای کارکرد درست آن، ساخت image و اجرای container لازم نیست.

هدف این طراحی این است که:
- نصب ماژول‌ها قابل تکرار شود،
- اجرای آن‌ها یکدست‌تر شود،
- و آپدیت آن‌ها از حالت دستی و مبهم خارج شود.

---

## محدوده

این سند شامل این موارد است:
- ورودی‌های لازم برای نصب ماژول
- مراحل بررسی و نصب
- مراحل اجرا
- ثبت ماژول در سیستم
- آپدیت نسخه جدید
- حالت‌های خطا و تصمیم‌های کنترلی

این سند شامل این موارد نیست:
- طراحی UI پنل
- طراحی Docker flow
- طراحی Registry برای imageها
- معماری کامل process manager
- جزئیات مانیتورینگ پیشرفته

---

## تعریف ماژول غیر داکری

ماژول غیر داکری ماژولی است که:
- استاتیک صرف نیست،
- ولی برای اجرا به container نیاز ندارد،
- و می‌تواند به‌صورت مستقیم روی runtime میزبان اجرا شود.

نمونه‌ها:
- سرویس Node.js
- برنامه Python
- worker پس‌زمینه
- ابزار CLI زمان‌بندی‌شده
- سرویس داخلی با پورت محلی

---

## پیش‌نیازهای بسته نصب

هر ماژول برای ورود به این flow باید حداقل این موارد را داشته باشد:

- فایل `module.json`
- فایل‌های پروژه
- فایل‌های نصب وابستگی متناسب با زبان، اگر لازم باشند
- نقطه شروع روشن برای اجرا
- نسخه مشخص

ورودی می‌تواند یکی از این دو حالت باشد:
1. بسته ZIP
2. مسیر محلی پروژه برای ابزار کمکی یا import محلی

---

## اجزای اصلی سیستم

### 1. ورودی نصب
مسئول دریافت بسته یا مسیر پروژه است.

وظایف:
- دریافت ZIP یا مسیر پوشه
- ایجاد job نصب
- ذخیره موقت اطلاعات اولیه

### 2. خواننده مشخصات ماژول
فایل `module.json` را می‌خواند و اعتبارسنجی اولیه انجام می‌دهد.

وظایف:
- بررسی وجود فایل
- parse کردن JSON
- بررسی فیلدهای پایه
- اعلام خطاهای ساختاری

### 3. اعتبارسنج نصب
تصمیم می‌گیرد آیا ماژول برای ادامه flow مناسب است یا نه.

وظایف:
- بررسی نوع ماژول
- بررسی وجود دستورهای لازم
- بررسی کامل بودن اطلاعات اجرا
- بررسی تداخل شناسه، نسخه و پورت

### 4. نصب‌کننده
وابستگی‌ها و build لازم را انجام می‌دهد.

وظایف:
- اجرای install
- اجرای build در صورت نیاز
- تولید لاگ نصب
- اعلام موفقیت یا شکست

### 5. ثبت‌کننده
اطلاعات نهایی ماژول را در مرجع داخلی CMS ثبت می‌کند.

وظایف:
- ثبت نسخه
- ثبت نوع اجرا
- ثبت وضعیت
- ثبت مسیر فایل‌ها
- ثبت اطلاعات لازم برای start و update

### 6. اجراکننده
ماژول را به‌عنوان یک سرویس عادی اجرا می‌کند.

وظایف:
- start کردن ماژول
- بررسی اولیه سلامت
- ثبت PID یا شناسه فرایند
- ثبت وضعیت running / stopped / failed

### 7. به‌روزرسان
وظیفه بررسی نسخه جدید و جایگزینی نسخه فعلی را دارد.

وظایف:
- مقایسه نسخه‌ها
- تشخیص تغییرات حساس
- توقف نسخه قبلی
- نصب نسخه جدید
- اجرای دوباره ماژول

---

## نقش `module.json`

فایل `module.json` شناسنامه فنی ماژول است و نقطه شروع این flow محسوب می‌شود.

این فایل باید اطلاعات لازم برای این تصمیم‌ها را فراهم کند:
- ماژول از چه نوعی است؟
- چگونه نصب می‌شود؟
- چگونه اجرا می‌شود؟
- آیا پورت لازم دارد؟
- چه متغیرهایی برای اجرا ضروری‌اند؟
- آیا نسخه جدید با قبلی هم‌خانواده است یا نه؟

سیستم نباید برای تصمیم‌های پایه به حدس زدن structure پروژه وابسته باشد.  
اگر اطلاعات لازم در `module.json` نباشد، نصب باید متوقف شود یا وارد حالت نیازمند بازبینی شود.

---

## مراحل اصلی نصب

### مرحله 1: دریافت ورودی
سیستم بسته ZIP یا مسیر محلی را دریافت می‌کند.

خروجی:
- job جدید نصب
- شناسه موقت job
- مسیر موقت استخراج یا بررسی

### مرحله 2: استخراج و آماده‌سازی
اگر ورودی ZIP باشد، فایل استخراج می‌شود.  
اگر ورودی مسیر باشد، ساختار پایه بررسی می‌شود.

بررسی‌ها:
- وجود ریشه مشخص
- نبود فایل‌های غیرمجاز واضح
- قابل خواندن بودن محتوا

### مرحله 3: خواندن `module.json`
سیستم فایل را از ریشه می‌خواند.

بررسی‌ها:
- فایل وجود دارد یا نه
- JSON معتبر است یا نه
- فیلدهای حداقلی کامل هستند یا نه

در صورت شکست:
- job باید متوقف شود
- خطا باید واضح و قابل فهم ثبت شود

### مرحله 4: اعتبارسنجی سازگاری
سیستم بررسی می‌کند:
- شناسه ماژول معتبر است یا نه
- نسخه معتبر است یا نه
- نوع ماژول قابل پشتیبانی است یا نه
- runtime شناخته‌شده است یا نه
- دستورهای لازم برای آن نوع ماژول وجود دارند یا نه

در این مرحله، ماژول یکی از این وضعیت‌ها را می‌گیرد:
- قابل نصب
- قابل نصب با هشدار
- نیازمند بازبینی
- رد شده

### مرحله 5: دریافت تنظیمات لازم
اگر ماژول به تنظیمات اجرا نیاز داشته باشد، سیستم باید آن‌ها را قبل از start شدن جمع‌آوری کند.

نمونه‌ها:
- متغیرهای محیطی لازم
- پورت پیشنهادی یا قطعی
- تنظیمات پایه اتصال

اگر اطلاعات ضروری موجود نباشد:
- نصب کامل نمی‌شود
- ماژول فقط در حالت آماده‌سازی باقی می‌ماند

### مرحله 6: نصب وابستگی‌ها
سیستم دستور نصب را اجرا می‌کند.

نمونه رفتار:
- اگر ماژول Node باشد، install وابستگی‌ها انجام می‌شود
- اگر Python باشد، dependency install انجام می‌شود
- اگر build لازم باشد، مرحله build بعد از install انجام می‌شود

خروجی:
- لاگ نصب
- وضعیت موفق / شکست
- زمان اجرا
- خطاهای احتمالی

### مرحله 7: ثبت در سیستم
بعد از نصب موفق، اطلاعات ماژول در مرجع داخلی ثبت می‌شود.

اطلاعات ثبت‌شونده:
- id
- name
- version
- type
- runtime
- وضعیت فعلی
- محل نصب
- تنظیمات اجرا
- زمان نصب
- منبع نصب

### مرحله 8: اجرای اولیه
اگر ماژول پروسه‌دار باشد، سیستم آن را start می‌کند.

خروجی:
- وضعیت اجرای اولیه
- نتیجه health check
- ثبت PID یا شناسه اجرای سرویس
- زمان start

### مرحله 9: نهایی‌سازی
ماژول به یکی از این وضعیت‌ها می‌رسد:
- Installed
- Running
- Stopped
- Failed
- Needs Review

---

## منطق اجرا

### انواع اجرای پشتیبانی‌شده

#### نوع 1: سرویس دارای پورت
ماژول به‌صورت یک سرویس محلی بالا می‌آید و روی یک پورت گوش می‌دهد.

نیازها:
- دستور start
- پورت معتبر
- health check یا راه ساده بررسی در دسترس بودن

#### نوع 2: worker بدون پورت
ماژول فقط به‌عنوان job runner یا سرویس پس‌زمینه اجرا می‌شود.

نیازها:
- دستور start
- عدم نیاز به endpoint عمومی
- روش ساده تشخیص alive بودن فرایند

#### نوع 3: اجرای دوره‌ای
ماژول باید در بازه‌های مشخص اجرا شود، نه به‌صورت دائم.

این نوع می‌تواند در فاز اول پشتیبانی محدود داشته باشد و در صورت نیاز به فاز بعد منتقل شود.

---

## سیاست پورت

برای ماژول‌های غیر داکری که پورت لازم دارند، سیستم باید یکی از این دو حالت را پشتیبانی کند:
1. پورت صریح از ماژول یا ادمین
2. تخصیص خودکار از بازه مجاز

قانون‌ها:
- پورت نباید با ماژول فعال دیگر تداخل داشته باشد
- پورت رزروشده باید در مرجع داخلی ثبت شود
- در زمان آپدیت، تغییر پورت باید تغییر حساس محسوب شود

---

## سیاست متغیرهای محیطی

ماژول می‌تواند لیستی از تنظیمات لازم داشته باشد.  
مقدار واقعی نباید در بسته نصب hardcode شود.

قوانین:
- فقط نام تنظیمات از روی `module.json` خوانده می‌شود
- مقدارها از پنل، تنظیمات سراسری یا secret store می‌آیند
- نبود مقدار ضروری مانع اجرای ماژول می‌شود
- تغییر env در آپدیت باید بررسی‌پذیر باشد

---

## مدل وضعیت ماژول

هر ماژول در هر لحظه یکی از این وضعیت‌ها را دارد:

- `draft`
- `validating`
- `installing`
- `installed`
- `starting`
- `running`
- `stopped`
- `failed`
- `needs_review`
- `updating`

این وضعیت‌ها باید هم در backend و هم در پنل قابل مشاهده باشند.

## ثبت خطا و لاگ

سیستم باید حداقل سه نوع لاگ نگه دارد:
- لاگ اعتبارسنجی
- لاگ نصب
- لاگ اجرای اولیه

هر خطا باید این اطلاعات را داشته باشد:
- مرحله‌ای که خطا رخ داده
- پیام قابل فهم
- جزئیات فنی برای تیم توسعه
- زمان رخداد

## منطق آپدیت

### ورودی آپدیت
نسخه جدید همان ماژول از یک بسته جدید یا منبع انتشار دریافت می‌شود.

### بررسی اولیه
سیستم باید این موارد را مقایسه کند:
- `id`
- `version`
- `type`
- `runtime`
- نیازهای env
- نیاز پورت
- دستور start

### انواع تغییر

#### تغییر عادی
مثال:
- افزایش نسخه
- اصلاحات داخلی
- تغییر توضیحات
- تغییر build بدون تغییر رفتار اجرایی

رفتار:
- آپدیت مستقیم با هشدار کم

#### تغییر حساس
مثال:
- تغییر نوع ماژول
- تغییر runtime
- تغییر نیاز پورت
- تغییر env ضروری
- تغییر الگوی اجرا

رفتار:
- توقف flow خودکار
- نیاز به بازبینی و تأیید

### مراحل آپدیت
1. دریافت نسخه جدید
2. اعتبارسنجی
3. مقایسه با نسخه فعلی
4. توقف سرویس قبلی
5. جایگزینی فایل‌ها
6. نصب وابستگی جدید
7. start مجدد
8. health check
9. ثبت نسخه جدید

---

## تصمیم‌های شکست و بازیابی

### خطا در اعتبارسنجی
نصب متوقف می‌شود و چیزی ثبت نهایی نمی‌شود.

### خطا در install
ماژول در وضعیت failed می‌رود و لاگ نصب حفظ می‌شود.

### خطا در start
نصب می‌تواند موفق ثبت شده باشد، ولی وضعیت نهایی failed یا stopped شود.

### خطا در update
اگر نسخه جدید بالا نیاید، سیستم باید حداقل:
- خطا را ثبت کند
- نسخه ناموفق را مشخص کند
- وضعیت ماژول را مبهم رها نکند

Rollback کامل می‌تواند در فاز بعدی اضافه شود.

---

## ملاحظات امنیتی پایه

- اجرای ماژول نباید با دسترسی بیش از حد انجام شود
- مسیر نصب هر ماژول باید جدا باشد
- secretها نباید در بسته نصب ذخیره شوند
- دستورهای install و start باید ثبت و قابل بازبینی باشند
- ماژول نباید بتواند تنظیمات ماژول دیگر را بازنویسی کند

---

## تصمیم‌های فاز اول

در فاز اول این تصمیم‌ها گرفته می‌شوند:
- `module.json` ورودی اصلی تشخیص ماژول است
- نصب از ZIP یا مسیر محلی پشتیبانی می‌شود
- پشتیبانی اصلی برای backend و worker است
- آپدیت خودکار فقط برای تغییرات غیرحساس انجام می‌شود
- rollback پیشرفته در فاز اول الزامی نیست

---

## موارد باز برای تصمیم بعدی

- process manager نهایی چه باشد
- محل دقیق ذخیره لاگ‌ها کجا باشد
- secret store داخلی باشد یا از تنظیمات فعلی استفاده شود
- پشتیبانی از jobهای زمان‌بندی‌شده در فاز اول بیاید یا نه
- آیا import از GitHub release در همین flow بیاید یا سند جدا بگیرد

---

## خروجی مورد انتظار این طراحی

بعد از پیاده‌سازی این design:
- توسعه‌دهنده می‌داند چه چیزی باید تحویل بدهد
- سیستم می‌داند با هر ماژول چه رفتاری داشته باشد
- ادمین می‌تواند با یک روند یکدست ماژول را نصب کند
- آپدیت ماژول‌ها از حالت موردی و پراکنده خارج می‌شود
