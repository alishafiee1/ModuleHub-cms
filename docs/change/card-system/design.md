# Design — Card System
**مسیر:** `docs/change/card-system/design.md`
**وابسته به:** `docs/change/card-system/proposal.md`
**وضعیت:** در حال طراحی

---

## بخش ۰ — پروژه در یه نگاه

ModuleHub یه سیستم مدیریت محتواست که بهش ماژول‌های مستقل نصب میشه — هر ماژول یه ZIP هست که شامل فرانت‌اند، بک‌اند و تنظیماته. صفحه اصلی CMS از کارت‌های این ماژول‌ها تشکیل میشه و ادمین می‌تونه همه‌چیز رو از یه داشبورد مدیریت کنه. الان پروژه تو مرحله MVP هست و این دیزاین مربوط به سیستم کارت‌هاست — یعنی همون قطعه‌هایی که اطلاعات هر ماژول رو روی صفحه نمایش میدن.

---

## بخش ۱ — ماجرا از چه قراره؟

یادته تو پروپوزال گفتیم کارت‌های ماژول‌ها باید سه حالت مختلف داشته باشن؟ یه حالت ساده برای وقتی دولوپر کارت سفارشی نساخته، یه حالت که ادمین بتونه ظاهرش رو تنظیم کنه، و یه حالت کامل برای وقتی دولوپر خودش HTML کارت رو نوشته. اینجا قراره بگیم این سه حالت دقیقاً با چه منطقی کار میکنن، از کجا داده می‌گیرن، و چطور با بقیه سیستم کنار هم قرار میگیرن.

---

## بخش ۲ — طرح چطوری کار میکنه؟

#### سه نوع کارت و منطق انتخاب بینشون

وقتی CMS میخواد کارت یه ماژول رو نشون بده، اول نگاه میکنه به `module.json` اون ماژول. اگه توش `card.hasCustomCard` برابر `true` بود، یعنی دولوپر یه کارت سفارشی آماده کرده — بریم سراغ حالت Developer. اگه نه، نگاه میکنه به `site-layout.json` و تنظیمات `appearance` اون ماژول. اگه ادمین mode رو روی `design` گذاشته بود، میریم حالت Design. و اگه هیچ‌کدوم نبود، حالت Standard نمایش داده میشه. این انتخاب هر بار که صفحه بارگذاری میشه اتفاق می‌افته — هیچ‌جایی کش نمیشه و هر تغییر در تنظیمات بلافاصله اثر داره.

#### حالت Standard — کارت پیش‌فرض

این ساده‌ترین حالته. CMS خودش یه کارت با اطلاعات پایه ماژول میسازه: اسم ماژول از `name` در module.json، آیکون از `meta.icon`، وضعیت اجرا (در حال اجرا / متوقف)، و یه دکمه Start/Stop اگه ماژول بک‌اند داشته باشه. ظاهر این کارت از تنظیمات `appearance` در site-layout.json خونده میشه — مثل رنگ پس‌زمینه، رنگ متن، و اندازه. اگه ادمین هیچ تنظیمی نذاشته باشه، مقادیر پیش‌فرض استفاده میشن. این حالت fallback نهایی سیستمه — یعنی حتی اگه همه چیز خراب شد، این کارت باید نمایش داده بشه.

#### حالت Design — HTML دلخواه ادمین

اینجا ادمین میتونه مستقیم HTML بنویسه و بگه کارتم اینطوری باشه. این HTML داخل یه `iframe` با sandbox محدود نمایش داده میشه — یعنی نه به داده‌های CMS دسترسی داره، نه میتونه اسکریپت خارجی لود کنه، نه میتونه با پنجره مادر حرف بزنه. این محدودیت‌ها عمدیه: ادمین ممکنه کد کامل ننویسه و نباید یه HTML ناقص بتونه کل صفحه رو خراب کنه. HTML فیلترشده با timestamp در `site-layout.json` ذخیره میشه — فیلد `appearance.htmlFilteredAt` این رو ثبت میکنه. اگه HTML خالی بود یا فیلتر مشکلی داشت، سیستم به حالت Standard برمیگرده.

#### حالت Developer — کارت سفارشی از ZIP

این قدرتمندترین حالته. دولوپر موقع ساخت ماژول، یه فایل `card.html` داخل ZIP گذاشته. موقع نصب، این فایل به مسیر `data/modules/[module-id]/card.html` کپی میشه. وقتی CMS میخواد این کارت رو نشون بده، محتوای فایل رو از طریق یه endpoint پروکسی (`/api/card-proxy/[moduleId]`) سرو میکنه و داخل یه iframe با `sandbox="allow-scripts allow-same-origin"` نمایش میده. دولوپر میتونه داخل این card.html با endpoint خود ماژول ارتباط برقرار کنه — مثلاً هر ۵ ثانیه داده تازه بگیره و نشون بده. endpoint پروکسی هم فقط برای ماژول‌هایی که `hasCustomCard: true` دارن فعاله.

#### ذخیره‌سازی تنظیمات کارت در site-layout.json

همه چیز مربوط به ظاهر کارت‌ها داخل `site-layout.json` نگه داشته میشه. هر کارت توی آرایه `cards` یه آبجکت داره که شامل `moduleId`، موقعیت (`position`)، اندازه (`w`, `h`)، breakpoint‌های موبایل و تبلت، و یه آبجکت `appearance` میشه. آبجکت appearance شامل `mode` (standard/design/developer)، `bgColor`، `textColor`، `icon`، و در حالت design فیلد `html` هست. هر بار که ادمین چیزی رو تغییر میده، این فایل آپدیت میشه و صفحه رفرش میکنه تا تغییر رو نشون بده.

#### پروکسی endpoint برای کارت‌های Developer

یه endpoint ساده داریم: `GET /api/card-proxy/[moduleId]`. این endpoint چک میکنه که ماژول واقعاً `hasCustomCard: true` داشته باشه، بعد فایل `card.html` رو از مسیر `data/modules/[moduleId]/card.html` میخونه و با `Content-Type: text/html` برمیگردونه. اگه فایل پیدا نشد یا ماژول `hasCustomCard` نداشت، یه HTML ساده با پیام خطا برمیگردونه — نه 404. این کار باعث میشه iframe هیچ‌وقت خالی نمونه.

---

## بخش ۳ — با چی میسازیمش؟

**`iframe` با sandbox:** یه تگ HTML استاندارد که محتوا رو داخل یه محیط ایزوله نشون میده.
**چرا انتخاب شد:** هم برای حالت Design و هم Developer نیاز داریم HTML دولوپر/ادمین بدون دسترسی به محیط اصلی CMS نمایش داشته باشه. iframe sandbox این کار رو بدون کتابخونه اضافه انجام میده.
**گزینه‌ای که رد شد:** Web Components — قابلیت‌هایش جالب بود ولی برای card.html های ساده‌ای که دولوپرها می‌نویسن زیادی پیچیده‌ست و shadow DOM کار با استایل‌ها رو سخت‌تر میکنه.

---

**`site-layout.json` برای ذخیره تنظیمات:** یه فایل JSON که همه چیز مربوط به چیدمان و ظاهر صفحه اصلی رو نگه میداره.
**چرا انتخاب شد:** این فایل از قبل در پروژه بود و ساختار v2 ش دقیقاً جای مناسبی برای appearance داره. نیاز به دیتابیس جدید یا endpoint جدید نیست.
**گزینه‌ای که رد شد:** دیتابیس SQLite — برای این حجم از تنظیمات که فقط ادمین میخونه و می‌نویسه، فایل JSON کافیه و overhead کمتری داره.

---

**فیلتر HTML برای حالت Design:** قبل از اینکه HTML ادمین ذخیره بشه، از یه فیلتر ساده رد میشه.
**چرا انتخاب شد:** ادمین ممکنه از اشتباه `<script src="...">` یا `onclick` بذاره که داخل iframe هم مشکل‌ساز میشه. فیلتر این موارد رو حذف میکنه قبل از ذخیره.
**گزینه‌ای که رد شد:** اعتماد کامل به sandbox — sandbox خوبه ولی یه لایه امنیتی اضافه‌تر هزینه‌ای نداره.

---

**polling ساده برای داده‌های live در کارت Developer:** card.html می‌تونه با `setInterval` هر چند ثانیه از endpoint ماژول داده بگیره.
**چرا انتخاب شد:** ساده‌ترین روشه و دولوپر ماژول خودش کنترل کامل داره.
**گزینه‌ای که رد شد:** WebSocket — برای یه کارت که شاید فقط دمای یه سنسور رو نشون میده زیادی heavy هست.

---

## بخش ۴ — جواب سوالایی که تو پروپوزال پرسیدیم

**اگه دولوپر هم card.html داشت هم ادمین Design mode انتخاب کرده بود، کدوم اولویت داشت؟**
Developer اولویت داره. منطق ساده‌ست: اگه `hasCustomCard: true` بود، دیگه به `appearance.mode` نگاه نمیکنیم. دولوپر زحمت کشیده کارت ساخته — احترامش رو نگه میداریم.

**اگه card.html دولوپر خراب بود یا خطا داشت؟**
iframe خطا رو داخل خودش نشون میده — بقیه صفحه سالم میمونه. CMS به طور خاص این خطا رو لاگ نمیکنه چون داخل sandbox اتفاق افتاده. مسئولیت درست بودن card.html با دولوپر ماژوله.

**حالت Design چقدر HTML میتونه داشته باشه؟**
محدودیت سایز خاصی نذاشتیم، ولی چون داخل یه کارت نمایش داده میشه، HTML های خیلی طولانی عملاً مشکل UI ایجاد میکنن. فعلاً این رو به خودآموز ادمین میسپاریم.

**آیا کارت Developer میتونه با سایر ماژول‌ها ارتباط برقرار کنه؟**
نه. card.html فقط میتونه با endpoint‌های ماژول خودش حرف بزنه. cross-module communication در این فاز نیست.

---

## بخش ۵ — چی قراره ساخته نشه؟

- **ویرایشگر بصری کارت:** ادمین HTML رو مستقیم تایپ میکنه — نه drag and drop، نه WYSIWYG. اینا برای بعده.
- **پیش‌نمایش live در حین ویرایش:** ادمین HTML رو ذخیره میکنه و بعد نتیجه رو میبینه — نه اینکه همزمان پیش‌نمایش داشته باشه.
- **کارت‌های چندصفحه‌ای یا قابلیت navigate:** کارت فقط اطلاعات نشون میده — نه router داره، نه صفحه‌بندی.
- **دسترسی کارت به session یا داده‌های کاربر CMS:** کارت‌ها کاملاً ایزوله‌ان از auth سیستم.
- **قابلیت resize کارت توسط کاربر عادی:** فقط ادمین میتونه اندازه کارت‌ها رو عوض کنه — از طریق Module Manager.
- **WebSocket یا real-time push از سمت CMS به کارت:** polling مسئولیت خود card.html دولوپره.

---

## بخش ۶ — ریسک‌ها و نکات مراقبتی

- **ماژولی که card.html نداره ولی `hasCustomCard: true` داره:** endpoint پروکسی باید graceful handle کنه و به جای 404 یه کارت fallback نشون بده. اگه این درست هندل نشه، صفحه اصلی برای اون کارت خالی میمونه.

- **HTML فیلتر نشده از ادمین:** اگه فیلتر یه حالت edge case رو miss کنه و یه اسکریپت مخرب ذخیره بشه، sandbox iframe باید جلوش رو بگیره. sandbox آخرین خط دفاعیه — فیلتر نباید تنها خط دفاع باشه.

- **کارت‌های سنگین که polling فراوان دارن:** اگه یه دولوپر card.html ای بنویسه که هر ۱۰۰ میلی‌ثانیه request میزنه، ماژولش زیر بار میره. این رو نمیشه از سمت CMS کنترل کرد — باید تو `dockerize-guide.md` بهش اشاره بشه.

- **تغییر module.json بعد از نصب:** اگه دولوپر `hasCustomCard` رو بعد از نصب دستی تغییر بده، سیستم تا restart یا reload بعدی تغییر رو نمیبینه چون module.json موقع نصب خونده میشه. باید مستند بشه که تغییر module.json بعد از نصب نیاز به reinstall داره.

- **تداخل استایل card.html با استایل CMS:** چون iframe `allow-same-origin` داره، ممکنه استایل‌های parent رو ببینه. دولوپرها باید بدونن استایل‌های خودشون رو explicit تعریف کنن.

---

## بخش ۷ — تست‌های تایید پیاده‌سازی

**تست ۱ — نمایش کارت Standard**
- **سناریو:** یه ماژول با `hasCustomCard: false` نصب شده و appearance در site-layout.json روی `standard` هست.
- **انتظار:** کارت با اسم و آیکون ماژول نمایش داده میشه. اگه ماژول بک‌اند داشته باشه دکمه Start/Stop هم هست.
- **روش تایید:** صفحه اصلی CMS رو باز کن — کارت باید فوری بدون loading طولانی ظاهر بشه.

**تست ۲ — نمایش کارت Design**
- **سناریو:** ادمین از Module Manager وارد تنظیمات یه ماژول شده، mode رو `design` انتخاب کرده و یه HTML ساده مثل `<div style="color:white">سلام</div>` ذخیره کرده.
- **انتظار:** کارت داخل iframe اون HTML رو نشون بده. اسکریپت اجرا نشه.
- **روش تایید:** در DevTools مرورگر، iframe رو inspect کن — sandbox attribute باید باشه و محتوا نمایش داده بشه.

**تست ۳ — نمایش کارت Developer**
- **سناریو:** ماژولی با `hasCustomCard: true` و فایل `card.html` نصب شده.
- **انتظار:** endpoint `/api/card-proxy/[moduleId]` وقتی call میشه فایل card.html رو برمیگردونه. کارت داخل iframe نمایش داده میشه.
- **روش تایید:** مستقیم endpoint رو در مرورگر باز کن — باید HTML ببینی نه 404.

**تست ۴ — fallback وقتی card.html وجود نداره**
- **سناریو:** ماژولی `hasCustomCard: true` داره ولی فایل `card.html` پیدا نمیشه (مثلاً حذف شده).
- **انتظار:** endpoint پروکسی یه HTML ساده با پیام خطا برمیگردونه — نه 404. بقیه صفحه سالم میمونه.
- **روش تایید:** فایل card.html یه ماژول Developer رو rename کن، صفحه رو رفرش کن — کارت‌های دیگه باید سالم باشن.

**تست ۵ — فیلتر HTML در حالت Design**
- **سناریو:** ادمین یه HTML با `<script src="https://evil.com/x.js"></script>` ذخیره میکنه.
- **انتظار:** بعد از ذخیره، وقتی کارت نمایش داده میشه، اون script tag نباید وجود داشته باشه.
- **روش تایید:** site-layout.json رو باز کن — فیلد `html` نباید شامل tag script باشه.

**تست ۶ — اولویت Developer روی Design**
- **سناریو:** یه ماژول هم `hasCustomCard: true` داره هم در site-layout.json مودش `design` هست.
- **انتظار:** کارت Developer نمایش داده میشه — نه حالت Design.
- **روش تایید:** inspect کن که iframe داره از `/api/card-proxy/` لود میکنه نه از HTML ذخیره‌شده.

---

وسلام — card system سه حالته آماده پیاده‌سازیه. هر جا ابهامی بود تو proposal هم هست، ولی اگه سوالی داشتی بگو.
