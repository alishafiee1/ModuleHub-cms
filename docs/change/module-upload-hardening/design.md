# طراحی — آپلود ماژول محکم‌تر

> این سند توضیح می‌دهد ModuleHub چطور ZIP ماژول را تمیزتر باز می‌کند، ریشهٔ محتوای ماژول را بهتر تشخیص می‌دهد، جلوی ترکیب اشتباه Docker را می‌گیرد، و پوشه‌های نیمه‌کاره را پاک می‌کند.  
> چرا این کار مهم است: [proposal.md](./proposal.md) · تیک‌های اجرا: [tasks.md](./tasks.md)

---

## تصویر کلی

فرض کن ادمین یک ZIP آپلود می‌کند. سیستم اول مطمئن می‌شود فایل واقعاً ZIP است و مسیر خطرناک داخلش نیست. بعد فایل‌ها را در پوشهٔ اختصاصی ماژول باز می‌کند. از اینجا به بعد این change چند رفتار تازه اضافه می‌کند: اگر ZIP یک پوشهٔ تکی داشته باشد، آن پوشه را یک سطح بالا می‌آورد؛ اگر سایت آماده داخل `dist` باشد، همان را به‌عنوان ریشهٔ قابل نمایش می‌شناسد؛ اگر Docker روشن باشد، اجازه نمی‌دهد بدون پروسه و پورت ثبت شود؛ و اگر ادمین وسط ویزارد منصرف شد، پوشهٔ ثبت‌نشده قابل پاکسازی است.

این یعنی ماژول مثل مهمانی وارد خانه می‌شود که هم در ورودی کنترل می‌شود، هم اگر چمدانش شکل رایج داشت سیستم کمکش می‌کند، هم اگر اصلاً وارد لیست مهمان‌ها نشد اتاقش خالی نمی‌ماند.

---

## محدوده

داخل این طراحی، چهار چیز را می‌سازیم: flatten برای ZIPهای تک‌پوشه‌ای، سرو `dist` برای SPA، اعتبارسنجی Docker در ویزارد و سرور، و پاکسازی آپلود رها شده.

خارج از این طراحی، کتابخانهٔ ماژول آماده، اجرای کامل PHP، docker-compose، و اعمال واقعی permissionهاست. این‌ها مهم‌اند، ولی اگر همه را قاطی همین تغییر کنیم، مسیر سادهٔ آپلود دوباره سنگین و گیج‌کننده می‌شود.

---

## فرایند اول — ZIP وقتی یک پوشهٔ اضافه دارد

خیلی از ZIPهایی که از GitHub یا ابزارهای build می‌آیند، همه‌چیز را داخل یک پوشه می‌گذارند؛ مثلاً `my-site/index.html`. CMS فعلی بعد از باز کردن ZIP دنبال `index.html` در ریشهٔ ماژول می‌گردد و آن را نمی‌بیند.

رفتار جدید این است: بعد از استخراج امن، سیستم نگاه می‌کند آیا همهٔ فایل‌های واقعی زیر یک پوشهٔ سطح اول هستند یا نه. اگر جواب بله بود، محتوای آن پوشه به ریشهٔ ماژول منتقل می‌شود. اگر ZIP چند پوشه یا فایل ریشه‌ای جدا داشت، سیستم دست نمی‌زند.

اگر این مرحله شکست بخورد، آپلود باید با پیام واضح fail شود و پوشهٔ نصفه‌کاره پاک شود؛ چون بدترین حالت این است که فایل‌ها ناقص بمانند و بعداً ماژول عجیب رفتار کند.

---

## فرایند دوم — تشخیص ریشهٔ قابل نمایش

برای سایت‌های Static، ریشهٔ قابل نمایش همان پوشهٔ ماژول است، به شرطی که `index.html` در آن باشد. اما برای SPAهای آماده، گاهی `index.html` داخل `dist` است.

رفتار جدید این است: هنگام سرو فایل، سیستم اول ریشهٔ ماژول را بررسی می‌کند. اگر `index.html` در ریشه بود، همان برنده است. اگر نبود و `dist/index.html` وجود داشت، `dist` ریشهٔ قابل نمایش می‌شود. بعد همهٔ مسیرهای فایل و fallback SPA از همان ریشه استفاده می‌کنند.

این تصمیم عمدی است: اگر سازنده در ریشه `index.html` گذاشته، یعنی احتمالاً خودش همان را می‌خواسته. پس `dist` فقط وقتی فعال می‌شود که ریشه فایل ورود نداشته باشد.

---

## فرایند سوم — Docker بدون پورت ذخیره نشود

Docker یعنی ماژول داخل یک کانتینر جدا اجرا می‌شود. کانتینر باید یک راه ارتباطی داشته باشد؛ در این پروژه آن راه همان پورت است. اگر Docker روشن باشد ولی `needsProcess` خاموش بماند، پورت صفر ذخیره می‌شود و اجرای Docker با نگاشت پورت صفر خراب می‌شود.

رفتار جدید دو لایه دارد. در مرورگر، وقتی ادمین Docker را روشن می‌کند، گزینهٔ «نیاز به پروسه» خودکار روشن و قفل می‌شود. در سرور هم دوباره همین قانون چک می‌شود، چون هیچ وقت نباید فقط به مرورگر اعتماد کنیم.

اگر درخواست اشتباه رسید، سرور با خطای قابل فهم جواب می‌دهد: Docker module needs a process and a port. این پیام برای ادمین یعنی «یا Docker را خاموش کن، یا بگذار ماژول با پروسه و پورت ثبت شود».

---

## فرایند چهارم — پاکسازی آپلود رها شده

آپلود ZIP قبل از پایان ویزارد انجام می‌شود. یعنی ممکن است فایل‌ها روی دیسک باشند ولی هنوز کارت در چیدمان سایت ساخته نشده باشد. اگر ادمین در همین فاصله cancel کند، پوشه یتیم می‌ماند.

رفتار جدید یک مسیر پاکسازی امن دارد: فقط Super Admin می‌تواند درخواست حذف بدهد، و سرور فقط وقتی پوشه را حذف می‌کند که شناسهٔ ماژول هنوز داخل `site-layout.json` ثبت نشده باشد. اگر ماژول ثبت شده باشد، این مسیر حق حذف ندارد و باید از حذف کامل ماژول استفاده شود.

برای نگهداری دستی هم یک اسکریپت سبک می‌گذاریم که پوشه‌های `mod-*` بدون ورودی layout را لیست کند و در حالت حذف، همان‌ها را پاک کند. این اسکریپت برای وقتی خوب است که قبلاً چند پوشهٔ یتیم ساخته شده‌اند.

## فازبندی پیاده‌سازی

### فاز ۱ — داک و آماده‌سازی

هدف این فاز این است که همه بدانند چرا این تغییر لازم است و چه چیزهایی داخل scope هستند. تموم شدن فاز یعنی proposal، design و tasks خوانا باشند و در tasks اصلی پروژه هم لینک داشته باشند.

### فاز ۲ — ZIP و `dist`

هدف این فاز این است که رایج‌ترین بسته‌بندی‌های ماژول بدون دستکاری دستی کار کنند. اول flatten پوشهٔ تکی می‌آید، بعد تشخیص `dist`. ریسک این فاز این است که سیستم زیادی باهوش شود و ساختار عمدی سازنده را خراب کند؛ برای همین فقط در حالت‌های ساده و قابل تشخیص رفتار خودکار داریم.

تموم شدن فاز یعنی تست واحد برای ZIP تک‌پوشه‌ای و تست سرو `dist` سبز باشد.

### فاز ۳ — ویزارد و پاکسازی

هدف این فاز این است که انتخاب اشتباه Docker از همان اول گرفته شود و آپلود رها شده روی دیسک نماند. وابستگی‌اش فاز ۲ نیست، ولی بهتر است بعد از آن بیاید تا مسیر upload کامل‌تر دیده شود.

تموم شدن فاز یعنی UI و سرور هر دو Docker بدون پروسه را رد کنند و cleanup فقط پوشهٔ ثبت‌نشده را حذف کند.

### فاز ۴ — Python

هدف این فاز این است که ماژول Python بعد از نصب وابستگی، راه شروع شدن هم داشته باشد. این فاز بعد از محکم شدن upload می‌آید، چون اول باید ورودی ماژول تمیز باشد، بعد سراغ runtime جدید برویم.

`E2E-UPLOAD-02`: ادمین ZIPی با `dist/index.html` آپلود می‌کند، ماژول را Start می‌زند، و مسیر `/modules/<id>/` صفحهٔ آماده را نشان می‌دهد.

`E2E-UPLOAD-03`: ادمین Docker را روشن می‌کند؛ UI اجازه نمی‌دهد گزینهٔ پروسه خاموش بماند و سرور هم درخواست اشتباه را رد می‌کند.

`E2E-UPLOAD-04`: ادمین بعد از upload از ویزارد خارج می‌شود؛ پوشهٔ ثبت‌نشده پاک می‌شود و در چیدمان سایت اثری از آن نیست.
