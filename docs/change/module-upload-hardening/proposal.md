# پروپزال — آپلود ماژول بی‌دردسرتر و مطمئن‌تر

> می‌خواهیم وقتی ادمین یک ZIP ماژول آپلود می‌کند، سیستم کمتر گیر بدهد، کمتر پوشهٔ نصفه‌کاره جا بگذارد، و برای سایت‌های Static، SPA، Docker و Python مسیر روشن‌تری داشته باشد.  
> طراحی: [design.md](./design.md) · کارها: [tasks.md](./tasks.md)

---

## بخش ۱ — خلاصه‌ی پروژه‌ای که بر ما گذشت

ModuleHub یه CMS ماژولاره؛ یعنی صفحهٔ اصلی مثل یه تابلو پر از کارت است و هر کارت، درِ ورودی یک ماژول جداست. ادمین به‌جای اینکه هر بار کد اصلی سایت را دستکاری کند، ZIP ماژول را آپلود می‌کند، چند سؤال ساده جواب می‌دهد، و کارت جدید روی صفحه می‌آید.

تا الان این مسیر برای سایت‌های سادهٔ HTML و خیلی از ماژول‌های Node خوب جواب داده. ولی بررسی واقعی پروژه نشان داد چند جای کار شبیه اسباب‌کشی نیمه‌کاره است: بعضی ZIPها پوشهٔ اضافه دارند، بعضی React/Vue فقط داخل `dist` آماده‌اند، Docker می‌تواند با یک انتخاب اشتباه پورت صفر بگیرد، و اگر ادمین وسط ویزارد انصراف بدهد پوشه‌ای روی دیسک می‌ماند که در نقشهٔ سایت ثبت نشده.

---

## بخش ۲ — ایده چی بود؟

**الف) خلاصه**

ایده اینه که مسیر آپلود ماژول را مثل پذیرش یک مهمان تازه مرتب کنیم. اگر مهمان با چمدان مرتب آمد، مستقیم اتاقش را پیدا کند. اگر چمدانش داخل یک کیف بزرگ‌تر بود، سیستم بفهمد و خودش یک لایه باز کند. اگر مهمان وسط کار رفت، اتاق نیمه‌کاره خالی نماند.

**ب) جونم برات بگه که:**

فرض کن سارا یک سایت React ساخته و خروجی آماده‌اش داخل پوشهٔ `dist` است. از نظر سارا سایت آماده است؛ ولی CMS دنبال `index.html` در ریشه می‌گردد و چیزی پیدا نمی‌کند. نتیجه؟ سارا فکر می‌کند CMS خراب است، در حالی که فقط شکل بسته‌بندی با چیزی که سیستم انتظار داشته فرق دارد.

یا فرض کن رضا ZIP را آپلود می‌کند، مرحلهٔ اول ویزارد را می‌بیند، بعد منصرف می‌شود. الان فایل‌ها روی دیسک می‌مانند ولی کارت ساخته نمی‌شود. مثل این است که یک اتاق در ساختمان ساخته‌ایم ولی هیچ دری در نقشه به آن وصل نیست. بعد از چند بار تکرار، سرور پر از پوشه‌های بی‌صاحب می‌شود.

می‌خواهیم این دردسرهای کوچک ولی اعصاب‌خردکن را جمع کنیم.

---

## بخش ۳ — چطور ممکنه؟

اول ZIP را هوشمندتر باز می‌کنیم. اگر همهٔ فایل‌ها داخل یک پوشهٔ تکی باشند، سیستم همان پوشه را یک سطح بالا می‌آورد تا `index.html` در جای قابل انتظار دیده شود. این کار فقط وقتی انجام می‌شود که واقعاً یک پوشهٔ تکی تمیز وجود داشته باشد، نه وقتی ساختار ZIP چندشاخه است.

بعد به CMS یاد می‌دهیم اگر در ریشهٔ ماژول `index.html` نبود ولی داخل `dist` بود، همان `dist` را مثل ریشهٔ سایت سرو کند. این برای پروژه‌های React و Vue خیلی طبیعی است.

برای Docker، ویزارد را از همان اول مراقب می‌کنیم. اگر ادمین Docker را روشن کرد، یعنی ماژول پروسه می‌خواهد و باید پورت داشته باشد. پس سیستم اجازه نمی‌دهد ترکیب اشتباه Docker روشن و پروسه خاموش ذخیره شود.

در آخر، برای آپلودهای رها شده یک راه تمیز می‌گذاریم: اگر ماژول هنوز در چیدمان ثبت نشده، بتوانیم پوشه‌اش را پاک کنیم. این کار جلوی همان پوشه‌های یتیم را می‌گیرد.

---

## بخش ۴ — خوبیش چیه؟

ادمین کمتر گیر می‌کند. سازندهٔ ماژول مجبور نیست هر بار حدس بزند ZIP را دقیقاً چطور بچیند. React/Vue آماده راحت‌تر نصب می‌شود. Docker با انتخاب اشتباه خراب نمی‌شود. سرور هم پر از پوشهٔ بی‌صاحب نمی‌ماند.

اقا نتیجه‌اش ساده است: مسیر اضافه کردن ماژول از «حواست باشه دقیقاً فلان‌طوری ZIP کنی» تبدیل می‌شود به «فایل آماده‌ات را بده، سیستم تا جای منطقی کمک می‌کند».

---

## بخش ۵ — چه پیدا کردم برات

بررسی کد فعلی نشان داد پایه‌های اصلی وجود دارند: استخراج ZIP امن است، path traversal رد می‌شود، SPA fallback برای مسیرهای عمیق داریم، و runtime ماژول‌های static/backend/docker جدا شده.

چیزی که کم است بیشتر از جنس تجربهٔ نصب است، نه ساختن کل موتور از صفر. یعنی لازم نیست ModuleHub را از نو بسازیم؛ باید مسیر ورود ماژول تازه را کمی باهوش‌تر و تمیزتر کنیم.

---

## بخش ۶ — ایده‌های جناب خودم یعنی AI

- یک گزارش کوچک بعد از upload نشان بدهیم: «ریشهٔ سایت از `dist` تشخیص داده شد» یا «پوشهٔ تکی ZIP باز شد». این باعث می‌شود ادمین بفهمد سیستم چه کمکی کرده.
- بعداً یک صفحهٔ «پوشه‌های رها شده» در تنظیمات بسازیم تا ادمین قبل از حذف، حجم و تاریخ هر مورد را ببیند.
- برای Python یک گزینهٔ ساده‌تر در ویزارد اضافه کنیم: «Flask / FastAPI» تا start command کمتر حدسی باشد.

(در این change فعلاً کتابخانهٔ ماژول آماده، docker-compose کامل، و sandbox واقعی permissionها را نمی‌سازیم؛ این‌ها فازهای جدا هستند.)

---

## بخش ۷ — سوالات و چالش‌ها

### ۱. اگر ZIP واقعاً چند پوشهٔ جدا داشت چی؟

- الف - دست نزنیم و همان ساختار را حفظ کنیم.
- ب - فقط وقتی همه‌چیز زیر یک پوشهٔ مشترک است flatten کنیم.
- انتخاب پیشنهادی: ب. چون کمک می‌کند، ولی ساختارهای پیچیده را خراب نمی‌کند.

### ۲. اگر هم ریشه `index.html` داشت، هم `dist/index.html`؟

- الف - ریشه برنده باشد، چون سازنده احتمالاً عمدی گذاشته.
- ب - همیشه `dist` را ترجیح بدهیم.
- انتخاب پیشنهادی: الف. چون کمترین غافلگیری را دارد.

### ۳. پاکسازی آپلود رها شده خطرناک نیست؟

فقط وقتی پوشه هنوز در `site-layout.json` ثبت نشده باشد اجازهٔ حذف می‌دهیم. یعنی اگر کارت ساخته شده، cleanup ساده حق ندارد آن را پاک کند.
