# پروپزال — وقتی کارت ماژول خودش یک ویترین کوچک است

> می‌خواهیم سازندهٔ هر ماژول بتواند روی صفحهٔ اصلی ModuleHub یک کارت اختصاصی بسازد — نه فقط آیکون و یک خط توضیح.  
> پوشه: `docs/change/card-view/` · جزئیات فنی: [design.md](./design.md) · راهنمای سازنده: [module-card-guide.md](./module-card-guide.md)

---

## بخش ۱ — خلاصه‌ی پروژه‌ای که بر ما گذشت

ModuleHub یه CMS ساده‌ست که سایت رو مثل یه تابلوی کارت‌کارت نشون میده. هر کارت یه ماژول مستقله — گالری، وبلاگ، داشبورد IoT، هر چی. ادمین ماژول رو آپلود می‌کنه، روی بوم جاش رو تنظیم می‌کنه، بازدیدکننده کلیک می‌کنه و میره داخل همون ماژول. بدون اینکه هر بار کد اصلی سایت دست بخوره.

**تا الان:** روی صفحهٔ اصلی هر ماژول فقط یه آیکون، یه عنوان و شاید یه خط توضیح داره. انگار مغازه ویترین نداره — داخلش چه خبره معلوم نیست. سازندهٔ ماژول هم ناامیده؛ محصولش نصفه دیده میشه. چند تا راهنمای قدیمی و پلن نیمه‌کاره هم هست که آدم رو گیج می‌کنه: کارت رو کجا بذارم؟ دادهٔ زنده از کجا بیاد؟ CMS وسط باشه یا نباشه؟

---

## بخش ۲ — ایده چی بود؟

**الف) خلاصه**

ایده اینه که **کارت روی بوم مال خود سازندهٔ ماژول باشه**. CMS فقط اون رو امن نشون بده و ادمین بتونه اندازه‌اش رو تنظیم کنه. سازنده خودش تصمیم می‌گیره داخل کارت چی باشه — ثابت، زنده، با دکمه، یا هر شکلی که به کارش میاد.

**ب) جونم برات بگه که:**

فرض کن سارا طراحه و یه ماژول پورتفولیو آپلود کرده. داخل ماژول عکس‌ها و نمونه‌کارهای قشنگی داره؛ ولی وقتی بازدیدکننده روی صفحهٔ اصلی نگاه می‌کنه، فقط یه آیکون خاکستری و یه عنوان می‌بینه. مشتری نمی‌فهمه داخل چه خبره و شاید اصلاً کلیک نکنه.

یا فرض کن رضا یه ماژول IoT ساخته که چراغ‌های خونه رو روشن و خاموش می‌کنه. می‌خواد روی همون کارتِ صفحهٔ اصلی ببینه «۱۲ دستگاه آنلاین‌ان» و یه دکمهٔ «برق اصلی» داشته باشه، بدون اینکه هر بار وارد کل پنل بشه. الان یا باید همه‌چیز رو توی توضیح مارک‌داون بنویسه که زنده نیست و عوض نمیشه، یا امیدوار باشه بعداً خود CMS براش چیزی بسازه.

می‌خوایم این شکاف رو ببندیم.

---

## بخش ۳ — چطور ممکنه؟

اول یه راهنمای عملی برای سازنده می‌نویسیم — [module-card-guide.md](./module-card-guide.md) — با سه داستان ساده: ماژول استاتیک، SPA، و ماژول backend مثل IoT. هر کدوم نشون میده سازنده چطور کارت خودش رو می‌سازه و اگه لازم بود دادهٔ زنده رو از کجا میاره.

بعد توی محصول: وقتی سازنده داخل ZIP ماژول یه فایل کارت گذاشته باشه، CMS همون رو روی صفحهٔ اصلی نشون میده — جدا و امن، که اگه مشکلی بود کل سایت نخوابه.

ادمین بعد از نصب ماژول — و اگه لازم بود Start زدن — میره «ویرایش چیدمان» و اندازهٔ کارت رو درست می‌کنه. اگه ماژول خاموش بود، بازدیدکننده کارت پیش‌فرض (آیکون و عنوان) رو می‌بینه — نه یه قاب خالی یا خطای عجیب.

سازنده داخل کارت هر کاری که لازم داره می‌کنه. CMS وسط ارسال و دریافت داده نمی‌ایسته؛ فقط کارت رو نشون میده. برای ماژول‌های backend مثل IoT، کد اصلی جای دیگه اجرا میشه — طبق سیاست [docker-module](../docker-module/proposal.md) — و کارت فقط خلاصهٔ همون سرویسه.

## بخش ۴ — خوبیش چیه؟

اقا چی بگم دیگه — وقتی هر سازنده می‌تونه ویترین خودش رو بسازه، صفحهٔ اصلی دیگه یه ردیف آیکون خاکستری نیست؛ هر کارت یه داستان کوچکه.

سازنده یه بار می‌خونه، ZIP رو درست می‌چینه، آپلود می‌کنه، تمام. نباید برای هر فیچر کوچک منتظر تصمیم تیم CMS بمونه یا بین سه راهنمای مختلف سرگردون بشه.

بازدیدکننده روی صفحهٔ اصلی خلاصهٔ قابل فهم می‌بینه. اگه خواست کلیک می‌کنه و میره داخل ماژول کامل — مثل همیشه.

اگه یه ماژول خراب شد، بقیهٔ کارت‌ها و کل صفحهٔ اصلی سالم می‌مونن. برای IoT هم دو سازنده می‌تونن دو روش مختلف برای دادهٔ زنده استفاده کنن و هر دو کار کنن — CMS بینشون فرق نمی‌ذاره.

## بخش ۵ — چه پیدا کردم برات (بررسی مشابه‌ها)

پروژهٔ دقیقاً مثل ModuleHub با «کارت سفارشی per-module» تو open source کم پیدا میشه — بیشتر CMSها یا ویجت ثابت میدن یا خودشون UI کارت رو می‌سازن.

نزدیک‌ترین‌ها اینا هستن:

- **وردپرس + پلاگین‌ها:** هر پلاگین توی داشبورد ویجت خودش رو داره، ولی روی صفحهٔ عمومی سایت معمولاً shortcode یا block ثابته — سازندهٔ ماژول مستقل اینقدر آزاد نیست.
- **Shopify App Store / npm package pages:** هر اپ یه کارت معرفی با preview داره — الهام برای «ویترین کوچک قبل از ورود»، ولی اونجا marketplace مرکزیه نه CMS شخصی.
- **Home Assistant Lovelace cards:** هر integration می‌تونه کارت UI خودش رو بسازه — از نظر «کارت = خلاصهٔ سرویس زنده» خیلی شبیهه، ولی اکوسیستم IoT محضه.

کد آمادهٔ copy-paste برای «کارت HTML داخل ZIP ماژول + sandbox امن» پیدا نکردم — باید خودمون قرارداد و لایهٔ امنیت رو بسازیم. ولی مسیر [docker-module](../docker-module/proposal.md) برای backend و [card-live-customization](../card-live-customization/proposal.md) برای دادهٔ زنده از طریق CMS، دو شاخهٔ جدا هستن که نباید قاطی بشن.

پلن iframe قدیمی رو هم [معلق کردیم](../../archive/1405-03-24-static-card-template-suspended/proposal.md) — این change جایگزینشه.

## بخش ۶ — ایده‌های جناب خودم یعنی AI

فک نکنم fable هم می‌تونست این همه ایده جذاب بده. ماشالا به خودم:

- **پیش‌نمایش زنده توی راهنما:** سازنده قبل از آپلود، توی یه صفحهٔ تست ببینه کارتش روی بوم چطور می‌شینه — مثل preview قبل از publish.
- **کارت‌های preset:** سه تا قالب آماده — «فقط متن»، «عکس + دکمه»، «عدد زنده + وضعیت» — که سازنده تازه‌کار سریع شروع کنه و بعد سفارشی کنه.
- **حالت fallback هوشمند:** اگه فایل کارت نبود یا parse نشد، CMS خودش از thumbnail و توضیح ماژول یه کارت قشنگ بسازه — نه آیکون خاکستری بی‌روح.
- **نشان «ماژول خاموش»:** وقتی backend IoT down باشه، کارت بگه «آفلاین» با رنگ ملایم — نه error قرمز ترسناک.

(چی **نمی‌سازیم** توی این change: درگاه مرکزی CMS که برای هر کارت مدام داده بگیره و بفرسته — اون مسیر جداست. validation کامل در هسته و skill نهایی برای سازنده رو هم به فاز بعد می‌سپاریم. ماژول رو مجبور نمی‌کنیم یه روش خاص برای داده انتخاب کنه.)

## بخش ۷ — سوالات و چالش‌ها

### ۱. اگه محتوای کارت آلوده یا باگ‌دار باشه چی میشه؟

- الف — کارت رو جدا از بقیهٔ صفحه نشون میدیم؛ کل سایت نمی‌خوابه
- ب — سازنده خودش مسئوله، CMS کاری نمی‌کنه
- پ — فقط ادمین می‌تونه کارت رو ببنده
- ج — نظر شما چیه؟

### ۲. اگه دو استاندارد موازی (iframe قدیمی + کارت جدید) هم‌زمان بمونن چی؟

- الف — سازنده دوباره گیج میشه؛ باید یکی رو رسماً جایگزین کنیم
- ب — هر دو باشن بهتره، آزادی بیشتره
- پ — فقط توی داک بنویسیم کدوم deprecated شده
- ج — نظر شما چیه؟

### ۳. ماژول backend بی‌حد منابع بخوره چی؟

- الف — داخل docker محدودش می‌کنیم؛ CMS وارد چرخهٔ داده نمیشه
- ب — CMS باید خودش rate limit بذاره
- پ — سازنده مسئوله، ما کاری نداریم
- ج — نظر شما چیه؟

### ۴. [module.json](../add%20manifest/design-module-json.md) و docker-module هنوز نیمه‌کاره‌ان — الان شروع کنیم یا صبر کنیم؟

- الف — الان شروع کنیم؛ توی design جا براشون بذاریم بدون قفل کردن سازنده
- ب — صبر کنیم تا manifest بیاد بعد کارت
- پ — فقط راهنما بنویسیم، کد محصول بعداً
- ج — نظر شما چیه؟
