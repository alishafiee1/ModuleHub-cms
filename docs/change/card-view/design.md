# طراحی — از ZIP تا کارت روی بوم

> **مسیر:** `docs/change/card-view/design.md`  
> **راهنمای سازنده:** [module-card-guide.md](./module-card-guide.md)

---

## این سند دربارهٔ چیه؟

از وقتی ZIP آپلود می‌شود تا وقتی بازدیدکننده روی بوم یک کارت سفارشی می‌بیند — **بدون** اینکه CMS وسط ارسال و دریافت دادهٔ ماژول بایستد.

---

## دو طرف میز — CMS و سازنده

فرض کن کارت مثل یک قاب عکس روی دیوار است. **CMS** قاب را می‌آویزد و اندازه می‌دهد. **سازندهٔ ماژول** عکس داخل قاب را می‌سازد و اگر عکس زنده است، خودش تصمیم می‌گیرد هر چند ثانیه عوض شود یا با WebSocket فوری به‌روز شود.

### CMS چه کار می‌کند

وقتی در ZIP فایل `card.html` (یا `card/index.html`) هست، CMS آن را از مسیر `/modules/<moduleId>/...` لود می‌کند و داخل **iframe جدا** روی بوم می‌گذارد. اگر HTML بد یا اسکریپت مشکل‌دار باشد، کل صفحهٔ اصلی نمی‌خوابد.

CMS اندازهٔ کارت را از `site-layout.json` می‌گیرد — همان `cardGrid` که ادمین در «ویرایش چیدمان» می‌کشد.

اگر ماژول **stopped** باشد، به‌جای iframe خراب یا خالی، همان کارت پیش‌فرض (آیکون + عنوان) را نشان می‌دهد.

CMS **هیچ سیاستی برای نحوهٔ ارسال/دریافت داده تعیین نمی‌کند**. فقط مسیر ماژول را باز می‌گذارد تا `card.html` با هر روشی که سازنده نوشته (HTTP، WebSocket، SSE، فایل ثابت، …) کار کند.

### سازندهٔ ماژول چه کار می‌کند

همهٔ منطق کارت — HTML، CSS، JavaScript، API، WebSocket، polling — **داخل ZIP و سرویس خود ماژول** است. CMS نمی‌گوید «هر پنج ثانیه GET بزن»؛ آن فقط یک مثال در راهنماست.

---

## فایل کارت کجاست؟

سازنده در ZIP یکی از این‌ها را می‌گذارد:

- `card.html` در ریشه — ترجیح اول  
- یا `card/index.html` اگر پوشهٔ جدا خواست  

اگر هیچ‌کدام نبود، مثل الان آیکون و عنوان پیش‌فرض می‌آید.

نام استاندارد **`card.html`** است (نه `cart.html`).

---

## سه نوع ماژول — به زبان ساده

### استاتیک

سارا فقط HTML و CSS می‌نویسد. شاید یک `fetch('./info.json')` برای یک عدد. شاید اصلاً JavaScript ندارد. **هیچ process سرور لازم نیست.** CMS فقط فایل را نشان می‌دهد.

### SPA

علی React build کرده. `index.html` اپ کامل است؛ `card.html` باید **سبک** بماند — نه کل bundle روی بوم. اگر عدد زنده می‌خواهد بدون backend خودش، یا API بیرونی (با CORS) یا نوع ماژول را backend کند.

### backend (مثل IoT)

رضا سرور دارد که با دستگاه‌ها حرف می‌زند. کارت خلاصهٔ همان سرویس است: «چند دستگاه آنلاین»، «یک دکمه».

**کد اصلی کجا اجرا می‌شود؟** در ماژول‌های backend طبق سیاست [docker-module](../docker-module/proposal.md) داخل محیط جدا (کانتینر). process ماژول جدا از هستهٔ CMS است؛ crash یا مصرف زیاد ماژول نباید سرور اصلی ModuleHub را از پا بیندازد.

**داده چطور به کارت می‌رسد؟** انتخاب سازنده — مثلاً:

- polling با `fetch` به `./api/card/summary`  
- WebSocket به `./api/card/ws`  
- SSE (Server-Sent Events)  
- ترکیب: WebSocket برای نمایش زنده، POST برای دکمه‌ها  

CMS فقط iframe را به `/modules/<moduleId>/card.html` وصل می‌کند؛ درخواست‌های داخل iframe به **همان origin ماژول** می‌روند، نه به API داخلی CMS.

**فرمان (دکمه، toggle):** سازنده endpoint خودش را می‌سازد (`POST`، WebSocket message، …) و خودش validate می‌کند. CMS وسط اعتبارسنجی نمی‌ایستد.

---

## چرا CMS وسط داده نمی‌ایستد؟

مدل قدیمی `card-live-customization` می‌خواست مرورگر فقط با `/api/card-ui` CMS حرف بزند و CMS به هر ماژول وصل شود. با زیاد شدن کارت‌ها، بار روی سرور اصلی زیاد می‌شود و هر باگ ماژول نزدیک‌تر به هسته می‌آید.

در `card-view` برعکس است: **ترافیک داده بین کارت و همان ماژول** است. CMS فقط صفحهٔ اصلی و چیدمان را نگه می‌دارد. این همان چیزی است که برای ماژول داکری هم معنی دارد — سرویس داخل کانتینر، کارت پنجرهٔ کوچک روی بوم.

---

## اندازهٔ کارت

ادمین ماژول را نصب می‌کند، در صورت نیاز Start می‌زند، می‌رود «ویرایش چیدمان» و گوشهٔ کارت را می‌کشد. مقدار در `site-layout.json` ذخیره می‌شود.

سازنده CSS را **انعطاف‌پذیر** بنویسد (`%`، `min-height`، فونت نسبی) چون ادمین ممکن است کارت را ۳×۳ یا ۶×۴ بگذارد.

---

## `module.json` — وقتی manifest بیاید

الان wizard هنوز کامل از [add manifest](../add%20manifest/design-module-json.md) پیروی نمی‌کند؛ پیش‌بینی می‌کنیم بخشی به نام `cardView`:

```json
"cardView": {
  "enabled": true,
  "template": "card.html"
}
```

فقط می‌گوید کارت سفارشی دارد یا نه — **نه** اینکه polling باشد یا WebSocket. آن جزئیات در راهنمای سازنده و کد خود ماژول است.

ماژول **docker** وقتی بیاید، همان قرارداد است — process داخل کانتینر، کارت همان `card.html` در ZIP.

---

## امنیت — خلاصه

HTML کارت فقط از ZIP می‌آید، نه از تنظیمات دستی ادمین روی بوم. iframe جدا می‌کند. برای backend، validate کردن actionها **داخل ماژول** است.

CMS نباید به خاطر باگ منطق یک ماژول down شود — ایزوله بودن process (به‌خصوص داکر) و عدم proxy کردن دادهٔ زنده از هسته، همین را هدف می‌گیرد.

## فرق با پلن‌های قبلی

پلن معلق [archive](../../archive/1405-03-24-static-card-template-suspended/proposal.md) می‌خواست کارت را بعد از فعال‌سازی **قفل** کند. اینجا ادمین می‌تواند اندازه را عوض کند.

`card-live-customization` درگاه CMS برای داده — برای static/spa لازم نیست؛ برای backend هم `card-view` اولویت دارد مگر جایی واقعاً به proxy CMS نیاز باشد (باید در آن change شفاف شود).
