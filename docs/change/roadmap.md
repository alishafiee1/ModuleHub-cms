# 🗺️ Roadmap — ModuleHub CMS
> **۵ فاز، ~۵۶ هفته** | هر فاز یه پلن مستقل و قابل تحویله

## 🟣 فاز ۱ — پایه‌گذاری (هفته ۱ تا ۱۳)
**هدف:** قبل از هر چیز، «شناسنامه ماژول» و «ساختار جدید داده‌ها» رو درست کنیم. این پایه همه چیزه — بدون این، هیچ‌کدوم از فازهای بعدی معنی ندارن.

***

### 🔵 پلن ۱-A — استاندارد `module.json` (هفته ۱ تا ۳)
**یه جمله:** تعریف کنیم هر ماژول «شناسنامه»اش دقیقاً چه شکلیه.

**کارهایی که باید انجام بشه:**
- ساخت `docs/standards/module-json-spec.md` — استاندارد کامل شناسنامه رو تعریف می‌کنه: چه فیلدهایی اجباریه، چه فیلدهایی اختیاریه، با مثال‌های واقعی
- ساخت `docs/standards/module-json-ai-guide.md` — این رو می‌دی به AI تا از روی کد ماژول، شناسنامه رو خودش بسازه. باید سؤال‌های راهنما داشته باشه: «آیا Dockerfile داره؟»، «فایل اصلی HTML کجاست؟»، «پورت چنده؟»

**ساختار `module.json`:**
```json
{
  "id": "my-module",
  "name": "نام ماژول",
  "version": "1.0.0",
  "type": "frontend-only | frontend-backend | backend-only",
  "install": {
    "hasBackend": false,
    "entryPoint": "index.html"
  },
  "backend": {
    "dockerComposePath": "backend/docker-compose.yml",
    "internalPort": 3000,
    "healthCheck": "/health"
  },
  "card": {
    "hasCustomCard": false,
    "cardPath": "card.html"
  },
  "meta": {
    "author": "نام سازنده",
    "icon": "assets/icon.png",
    "tags": ["iot", "dashboard"]
  }
}
```

**نتیجه قابل اندازه‌گیری:** یه ماژول نمونه بساز و کدش رو به AI بده — AI باید بدون کمک بتونه `module.json` رو بسازه.

***

### 🔵 پلن ۱-B — اصلاح داکیومنت‌های اصلی (هفته ۴ تا ۵)
**یه جمله:** تناقض بین «wizard سؤال‌محور قدیمی» و «manifest اجباری جدید» رو از توی داک‌ها پاک کنیم.

**کارهایی که باید انجام بشه:**
- در `docs/proposal.md` بخش نصب ماژول رو اصلاح کن — جمله‌هایی که از wizard و فرم سؤال حرف می‌زنن حذف بشن، جاشون بنویس: «CMS اول `module.json` رو چک می‌کنه، اگه نبود خطا میده و راهنما نشون میده»
- در `docs/architecture-and-structure.md` ساختار درختی ZIP رو آپدیت کن تا `module.json` به عنوان فایل اجباری توش باشه:
```
module-name.zip
├── module.json        ← (اجباری) شناسنامه
├── frontend/
│   └── index.html
├── backend/           ← (اختیاری، فقط با Docker)
│   └── docker-compose.yml
└── card.html          ← (اختیاری) کارت سفارشی
```
- در `docs/change/add manifest/proposal.md` بخش معیار موفقیت رو آپدیت کن — اضافه کن که راهنمای AI-Ready هم باید وجود داشته باشه

**نتیجه قابل اندازه‌گیری:** هیچ جایی توی داک‌های اصلی از wizard یا فرم سؤال‌محور حرف نزنه.

***

### 🔵 پلن ۱-C — تغییر Upload Handler (هفته ۶ تا ۸)
**یه جمله:** کد CMS رو عوض کنیم تا وقتی ZIP میاد، اول دنبال `module.json` بگرده.

**کارهایی که باید انجام بشه:**
- فایل upload handler در کد CMS رو پیدا کن (جایی که ZIP رو باز می‌کنه)
- قبل از هر کاری چک کن `module.json` داخل ZIP هست؟
  - اگه **هست:** اطلاعات رو از اون بخون، wizard رو کاملاً حذف کن
  - اگه **نیست:** نصب رو متوقف کن و این پیام رو نشون بده: «این ماژول فایل شناسنامه (`module.json`) ندارد. راهنمای ساخت آن را ببینید → [لینک]»
- wizard قدیمی (فرم سؤال‌محور) کاملاً حذف بشه — دیگه لازم نیست
- اطلاعات ماژول از `module.json` خونده بشه و در `data/modules/[module-id]/module.json` ذخیره بشه

**نتیجه قابل اندازه‌گیری:** ZIP بدون `module.json` → پیام خطا + لینک راهنما. ZIP با `module.json` → نصب موفق بدون هیچ سؤالی.

***

### 🔵 پلن ۱-D — ساختار جدید داده‌ها (هفته ۹ تا ۱۳)
**یه جمله:** چون سایت هنوز آزمایشیه، ساختار `site-layout.json` رو از نو و تمیز طراحی می‌کنیم — نه migration، بلکه reset کامل.

**وضعیت قدیمی (الان):**
همه چیز — چیدمان کارت‌ها، تنظیمات ماژول، پورت، اطلاعات — همه توی یه فایل `site-layout.json` ریخته.

**ساختار جدید:**
```
data/
├── site-layout.json        ← فقط چیدمان بوم (موقعیت و اندازه کارت‌ها)
└── modules/
    ├── module-id-1/
    │   └── module.json     ← شناسنامه + تنظیمات این ماژول
    └── module-id-2/
        └── module.json
```

**فرمت `site-layout.json` جدید:**
```json
{
  "version": 2,
  "grid": { "columns": 12 },
  "cards": [
    {
      "moduleId": "my-module",
      "position": { "x": 0, "y": 0, "w": 2, "h": 2 },
      "breakpoints": {
        "mobile": { "x": 0, "y": 0, "w": 4, "h": 2 },
        "tablet": { "x": 0, "y": 0, "w": 3, "h": 2 }
      },
      "appearance": {
        "mode": "standard",
        "bgColor": "#1a1a2e",
        "textColor": "#ffffff",
        "icon": "solar:home-bold"
      }
    }
  ]
}
```

**کارهایی که باید انجام بشه:**
- فایل `site-layout.json` قدیمی رو پاک کن، با فرمت v2 از نو بساز
- کد CMS که این فایل رو می‌خونه آپدیت کن تا با فرمت جدید کار کنه
- پوشه `data/modules/` رو بساز — هر ماژول موجود رو از نو با `module.json` نصب کن

**نتیجه قابل اندازه‌گیری:** CMS با ساختار جدید بالا میاد، هیچ داده قدیمی نیست که مشکل ایجاد کنه.

## 🟠 فاز ۲ — Docker Policy (هفته ۱۴ تا ۲۴)
**هدف:** هیچ بک‌اندی مستقیم روی سرور اجرا نشه. همه باید داخل «جعبه امن» Docker باشن. کد قدیمی که با `systemd-run` کار می‌کنه کنار میره.

***

### 🟠 پلن ۲-A — پروپزال رسمی Docker Policy (هفته ۱۴ تا ۱۵)
**یه جمله:** اول روی کاغذ مشخص کنیم قانون دقیقاً چیه، بعد بریم سراغ کد.

**کارهایی که باید انجام بشه:**
- فایل `docs/change/docker-module-policy/proposal.md` رو بساز با این محتوا:
  - چرا Docker؟ (امنیت، پاکیزگی، قابل پیش‌بینی بودن، کنترل منابع)
  - جریان دقیق نصب ماژول با بک‌اند (دیاگرام تصمیم‌گیری)
  - متن دقیق پیام راهنما برای کاربر
  - لیست فایل‌های CMS که باید تغییر کنن
- در `docs/proposal.md` بخش «اجرای بک‌اند» رو اصلاح کن — جمله‌های `npm install` مستقیم و `systemd-run` رو حذف کن
- در `docs/architecture-and-structure.md` اضافه کن: «بک‌اند فقط از طریق Docker Compose مدیریت میشه»

**نتیجه قابل اندازه‌گیری:** هیچ جایی توی داک‌ها از اجرای مستقیم بک‌اند حرف نزنه.

***

### 🟠 پلن ۲-B — پیام راهنما برای ماژول بدون Docker (هفته ۱۶ تا ۱۷)
**یه جمله:** وقتی کسی ZIP با بک‌اند ولی بدون Docker می‌فرسته، یه پیام کامل راهنما نشون بده.

**کارهایی که باید انجام بشه:**
- در upload handler (جایی که `module.json` رو می‌خونیم) این منطق رو اضافه کن:
  - اگه `hasBackend: true` ولی `dockerComposePath` خالیه → نصب متوقف بشه
  - این پیام نشون داده بشه:
    > *«⚠️ این ماژول کد بک‌اند دارد. برای نصب امن، ابتدا آن را داخل Docker قرار دهید، سپس دوباره آپلود کنید. راهنما را بخوانید یا به AI بدهید تا این کار را انجام دهد.»*
- یه صفحه راهنمای داکرایز کردن ماژول بساز (`docs/standards/dockerize-guide.md`) که توضیح بده:
  - یه `Dockerfile` ساده چه شکلیه
  - یه `docker-compose.yml` ساده چه شکلیه
  - چطور `module.json` رو برای Docker تنظیم کن

**نتیجه قابل اندازه‌گیری:** ZIP با بک‌اند ولی بدون Docker → پیام راهنما نشون داده میشه و نصب نمیشه.

***

### 🟠 پلن ۲-C — جایگزینی Start/Stop با Docker (هفته ۱۸ تا ۲۱)
**یه جمله:** کد اجرا کردن ماژول رو از «مستقیم» به «از طریق Docker» تغییر بدیم.

**کارهایی که باید انجام بشه:**
- فایل کد مربوط به Start/Stop ماژول رو پیدا کن
- کد `systemd-run` و اجرای مستقیم Node/Python رو حذف کن
- جاشون:
  - `Start` → `docker compose -f [مسیر] up -d`
  - `Stop` → `docker compose -f [مسیر] down`
  - `Restart` → `docker compose -f [مسیر] restart`
  - `Logs` → `docker logs [container-name]` به جای خواندن فایل لاگ مستقیم
- ماژول‌های `frontend-only` دست نخورن — اونا بک‌اند ندارن و اجرایی ندارن

**نتیجه قابل اندازه‌گیری:** ماژول Docker دار: Start → container بالا میاد. Stop → container میره. لاگ از Docker نشون داده میشه.

***

### 🟠 پلن ۲-D — پنل تأیید تنظیمات Docker (هفته ۲۲ تا ۲۴)
**یه جمله:** ادمین باید بتونه تنظیمات Docker ماژول رو از پنل ببینه و در صورت نیاز تغییر بده.

**کارهایی که باید انجام بشه:**
- در صفحه مدیریت هر ماژول (⚙) یه بخش جدید «تنظیمات Docker» اضافه کن
- توی این بخش:
  - محدودیت CPU (مثلاً `0.5` = نصف یه هسته)
  - محدودیت RAM (مثلاً `256m`)
  - پورت‌هایی که container نشون میده
  - وضعیت container: ✅ روشن | ⛔ خاموش | ⚠️ ارور
  - Environment Variables — نشون داده بشن ولی مقادیر حساس (رمزها) با `***` پوشیده باشن
- این تنظیمات تأیید یا اصلاح ادمین رو می‌خوان قبل از اجرا

**نتیجه قابل اندازه‌گیری:** ادمین بدون دست زدن به سرور می‌تونه CPU و RAM هر ماژول رو محدود کنه.

***
***

## 🟢 فاز ۳ — سیستم کارت یکپارچه (هفته ۲۵ تا ۳۷)
**هدف:** کارت‌ها دیگه فقط «اسم + آیکون» نباشن. سه سطح سفارشی‌سازی داشته باشیم که هر کس به اندازه دانشش ازشون استفاده کنه.

***

### 🟢 پلن ۳-A — پروپزال یکپارچه کارت (هفته ۲۵ تا ۲۶)
**یه جمله:** دو پروپزال قدیمی رو بازنشسته کن، یه پروپزال جدید بنویس.

**کارهایی که باید انجام بشه:**
- فایل `docs/change/card-system/proposal.md` رو بساز — سه حالت کارت رو با جزئیات کامل توضیح بده
- در بالای `docs/change/card-view/proposal.md` بنویس:
  > `⚠️ این پروپزال منسوخ شده. به docs/change/card-system/proposal.md مراجعه کنید`
- همین کار رو برای `docs/change/card-live-customization/proposal.md` انجام بده

**نتیجه قابل اندازه‌گیری:** یه پروپزال واحد که هر سه حالت رو با جزئیات توضیح میده.

***

### 🟢 پلن ۳-B — حالت Standard کارت (هفته ۲۷ تا ۲۹)
**یه جمله:** ادمین بتونه از پنل، رنگ و متن و آیکون کارت رو عوض کنه — بدون هیچ دانش فنی.

**کارهایی که باید انجام بشه:**
- در صفحه مدیریت ماژول (⚙) یه تب جدید «ظاهر کارت» اضافه کن
- توی این تب قابل تنظیم باشه:
  - انتخاب رنگ پس‌زمینه (color picker)
  - انتخاب رنگ متن
  - آپلود آیکون یا انتخاب از کتابخانه آیکون
  - آپلود عکس پس‌زمینه
  - متن توضیح با Markdown ساده (برای **بولد** و `کد` و لینک)
- تغییرات رو در `site-layout.json` ذخیره کن زیر `appearance.mode: "standard"`
- پیش‌نمایش زنده: همون‌طور که تنظیم می‌کنی، کارت بلافاصله عوض میشه

**نتیجه قابل اندازه‌گیری:** ادمین بدون لمس کد می‌تونه کارت هر ماژول رو کاملاً شخصی‌سازی کنه.

***

### 🟢 پلن ۳-C — حالت Design کارت (هفته ۳۰ تا ۳۳)
**یه جمله:** کاربر می‌تونه یه HTML/CSS/JS دستی paste کنه و کارت دقیقاً اون رو بدون هیچ حاشیه‌ای نشون بده.

**کارهایی که باید انجام بشه:**
- در تب «ظاهر کارت» یه گزینه «حالت طراح» اضافه کن
- یه textarea بزرگ نشون بده که کاربر HTML می‌نویسه یا paste می‌کنه
- قبل از ذخیره، HTML از یه **فیلتر امنیتی** رد بشه که این‌ها رو حذف کنه:
  - `<script>` که به URL خارجی وصله
  - `<iframe>` با src خارجی
  - دسترسی به `document.cookie` یا `window.parent`
  - `fetch` یا `XMLHttpRequest` به آدرس‌های خارجی
- HTML فیلترشده داخل یه `<iframe sandbox>` محدود اجرا بشه (داخل یه قفس — نمی‌تونه به بقیه صفحه دسترسی داشته باشه)
- تاریخ فیلترشدن ذخیره بشه (`htmlFilteredAt`)

**نتیجه قابل اندازه‌گیری:** HTML مخرب فیلتر میشه. HTML سالم کامل نشون داده میشه.

***

### 🟢 پلن ۳-D — حالت Developer کارت (هفته ۳۴ تا ۳۷)
**یه جمله:** سازنده ماژول می‌تونه یه کارت کاملاً سفارشی با داده زنده داخل ZIP بذاره.

**کارهایی که باید انجام بشه:**
- وقتی ZIP آپلود میشه و `card.html` داخلش بود:
  - استخراج و ذخیره کن در `data/modules/[id]/card.html`
  - در `module.json` ثبت کن: `card.hasCustomCard: true`
- یه endpoint proxy بساز: `/api/card-proxy/[moduleId]/` — مرورگر از طریق این با ماژول حرف می‌زنه نه مستقیم
- `card.html` داخل `<iframe sandbox="allow-scripts allow-same-origin">` نشون داده بشه
- کارت می‌تونه هر چند ثانیه یکبار از endpoint ماژول داده بگیره و خودش آپدیت بشه (polling)

**اولویت نمایش (منطق سیستم):**
```
Developer card داریم؟ → نشون بده
          ↓ نه
Design HTML داریم؟ → نشون بده
          ↓ نه
Standard settings → نشون بده
          ↓ نه
Fallback: آیکون + نام ماژول
```

**نتیجه قابل اندازه‌گیری:** ماژول IoT می‌تونه کارتش رو هر ۵ ثانیه با داده زنده (مثلاً دما) آپدیت کنه.

***
***

## 🔴 فاز ۴ — نقش‌ها و سطوح دسترسی (هفته ۳۸ تا ۴۶)
**هدف:** الان یا «همه‌کاره» هستی یا «هیچ‌کاره». باید سطح‌های مختلف اضافه کنیم.

***

### 🔴 پلن ۴-A — طراحی سیستم نقش‌ها (هفته ۳۸ تا ۳۹)
**یه جمله:** اول کاغذی تعریف کنیم هر نقش دقیقاً چه کاری می‌تونه انجام بده.

**کارهایی که باید انجام بشه:**
- `docs/change/role-based-access/proposal.md` رو کامل بنویس — چهار نقش تعریف کن:

| نقش | توضیح ساده | مثال |
|-----|-----------|------|
| **Super Admin** | همه‌کاره — نصب، حذف، تنظیمات کل سیستم | صاحب سرور |
| **Admin** | مدیریت ماژول‌ها — Start/Stop، ظاهر، بکاپ | مدیر داشبورد |
| **Viewer** | فقط می‌تونه کارت‌ها رو ببینه — هیچ تغییری نمیده | مهمان |
| **Module User** | فقط با یه ماژول خاص کار می‌کنه | کاربر یه سرویس |

- جدول کامل «هر نقش چه API‌هایی رو می‌تونه صدا بزنه» رو بنویس

**نتیجه قابل اندازه‌گیری:** جدول کامل مجوزها برای هر نقش آماده‌ست.

***

### 🔴 پلن ۴-B — پیاده‌سازی نقش‌ها در Backend (هفته ۴۰ تا ۴۳)
**یه جمله:** هر API که CMS داره باید نقش کاربر رو چک کنه قبل از انجام کار.

**کارهایی که باید انجام بشه:**
- سیستم session موجود رو گسترش بده تا نقش رو هم توی «بلیت دیجیتال» ذخیره کنه
- برای هر endpoint حساس (نصب، حذف، Start/Stop) یه middleware چک اضافه کن: «آیا این کاربر اجازه داره؟»
- اگه اجازه نداشت → پیام «شما دسترسی لازم را ندارید» (کد HTTP 403)
- Super Admin بتونه نقش بقیه رو از پنل مدیریت کنه

**نتیجه قابل اندازه‌گیری:** کاربر Viewer نمی‌تونه دکمه Start رو بزنه — نه از UI، نه مستقیم از API.

***

### 🔴 پلن ۴-C — UI مدیریت کاربران (هفته ۴۴ تا ۴۶)
**یه جمله:** Super Admin باید از پنل بتونه کاربر اضافه کنه، نقش بده، و حذف کنه.

**کارهایی که باید انجام بشه:**
- یه صفحه «مدیریت کاربران» اضافه کن (فقط Super Admin می‌بینتش)
- توی این صفحه:
  - لیست همه کاربران با نقش‌هاشون
  - دکمه «کاربر جدید» — اسم، رمز، نقش
  - دکمه تغییر نقش
  - دکمه حذف کاربر
- دکمه‌هایی که کاربر به اونها دسترسی نداره، توی UI هم مخفی یا غیرفعال باشن

**نتیجه قابل اندازه‌گیری:** Super Admin می‌تونه یه Viewer بسازه که فقط داشبورد رو ببینه و هیچ چیزی رو تغییر نده.
