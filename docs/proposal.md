# ModuleHub CMS

## سامانه‌ای که بدون دست زدن به کد اصلی سایت، هر قابلیتی را اضافه می‌کند

---

## این پروژه برای کیست؟

شما یک سایت دارید. می‌خواهید به آن گالری عکس، وبلاگ آموزشی، داشبورد زنده، فروشگاه کوچک، یا هر چیز دیگری اضافه کنید.

**مشکل قدیمی:** باید بروید کد سایت را باز کنید، فایل جدید بسازید، مسیردهی دستی اضافه کنید، وای به حال روزی که بخواهید یک قابلیت را حذف کنید!

**راه حل ModuleHub:** سایت شما فقط یک صفحه اصلی با کلی کارت است. هر کارت = یک ماژول مستقل. ماژول جدید = یک کارت جدید. بدون دست زدن به کد اصلی.

---

## ساختار ساده

```
خانه سایت (صفحه اصلی)
├── کارت گالری عکس → یک ماژول کامل
├── کارت آموزش مارک‌داون → یک ماژول دیگر
├── کارت آمار زنده → ماژول بعدی
└── کارت + → برای اضافه کردن ماژول جدید (فقط ادمین می‌بیند)
```

هر کارت که بزنید، وارد همان ماژول می‌شوید. مثل این است که چندین سایت کوچک داخل یک سایت دارید.

---

## پوشه‌ها فقط روی کاغذ هستند

می‌خواهید نمونه‌کارهایتان را دسته‌بندی کنید؟ یک پوشه مجازی می‌سازید.

- پوشه فقط در یک فایل JSON تعریف می‌شود (همان `site-layout.json`)
- روی سرور هیچ پوشه فیزیکی ساخته نمی‌شود
- فقط برای مرتب‌سازی کارت‌ها در صفحه اصلی است

مثل این می‌ماند که در فایل اکسپلورر یک پوشه جدید بسازید، اما در واقعیت هارددیسک شما چیزی عوض نمی‌شود.

---

## ادمین چه کارهایی می‌کند؟

### اضافه کردن ماژول جدید

در انتهای صفحه، یک کارت با علامت **+** می‌بینید (فقط وقتی وارد شده‌اید).

روی آن کلیک کنید، سه راه دارید:

| گزینه | چه می‌شود |
|-------|-----------|
| **پوشه جدید** | فقط یک دسته در JSON ساخته می‌شود |
| **آپلود فایل ZIP** | ماژول شخصی خود را نصب می‌کنید |
| **از کتابخانه** | ماژول آماده (مثل گالری) را انتخاب می‌کنید |

---

### نصب ماژول با ZIP — ساده‌ترین روش

**قدم ۱ — آماده‌سازی:** پروژه خود را در یک پوشه جمع می‌کنید. هر چیزی می‌تواند باشد: چند فایل HTML، یک برنامه React، یک اپ Node.js، یا حتی یک پروژه با Docker.

> **نکته وابستگی‌ها:** اگر ماژول شما به کتابخانه‌های خاصی نیاز دارد (مثل Flask، Django، React)، بهتر است فایل `requirements.txt` یا `package.json` را در ریشه ZIP قرار دهید. سیستم این فایل‌ها را شناسایی کرده و به‌طور خودکار وابستگی‌ها را نصب می‌کند. برای ماژول‌های Dockerized، همه چیز داخل کانتینر انجام می‌شود.

**قدم ۲ — آپلود:** فایل ZIP را آپلود می‌کنید.

**قدم ۳ — پاسخگویی به سه سوال ساده:**

| سوال | مثال پاسخ |
|------|-----------|
| آیا در Docker اجرا شود؟ | بله / خیر |
| پورت داخلی چیست؟ | ۳۰۰۰ (برای Node.js) یا **خالی** (برای HTML/CSS ساده) |
| چه دسترسی‌هایی نیاز دارد؟ | شبکه، فایل، یا هیچ‌کدام |

> **پورت داخلی:** اگر ماژول **سرور** دارد (Node، Python، …) پورت را وارد کنید. اگر سایت **آماده HTML/CSS** یا فایل `dist/` است، خالی بگذارید — CMS بدون پروسه جدا سرو می‌کند (جزئیات: `module-hosting-guide.md`).

**قدم ۴ — تمام:** سیستم خودش فایل‌ها را در جای درست قرار می‌دهد، تنظیمات را ذخیره می‌کند، و ماژول را اجرا می‌کند.

> **نکته مهم:** شما دیگر احتیاجی ندارید فایل `manifest.json` دستی بسازید. سیستم بعد از آپلود، فرم را نشان می‌دهد و خودش فایل تنظیمات را در `site-layout.json` ذخیره می‌کند.

### مدیریت ماژول‌ها

روی هر کارت، یک چرخ‌دنده ⚙ وجود دارد. با کلیک روی آن:

**وضعیت ماژول** یکی از این سه حالت است:

| وضعیت | معنی | اقدام پیشنهادی |
|-------|------|----------------|
| `running` | ماژول در حال اجراست | — |
| `stopped` | توسط ادمین متوقف شده | دکمه Start |
| `crashed` | به‌دلیل کمبود RAM (OOM Kill) یا خطای داخلی از کار افتاده | لاگ را ببینید، منابع را افزایش دهید، دوباره Start کنید |

- وضعیت را ببینید (با برچسب و رنگ مشخص در کارت)
- ماژول را استارت یا استاپ کنید
- از GitHub به‌روزرسانی بگیرید
- تنظیمات (پورت، منابع، رمز) را عوض کنید
- **مشاهده لاگ خطاها** (در بخش لاگ متمرکز)
- ماژول را حذف کنید

اگر برای ماژول رمز جداگانه گذاشته باشید، حتی بدون لاگین به عنوان ادمین اصلی سایت هم می‌توانید آن را مدیریت کنید.

---

### محدودیت منابع (CPU و RAM) — چرا و چگونه؟

روی یک سرور واقعی، ماژول‌ها نباید کل منابع را مصرف کنند. ModuleHub از قابلیت‌های بومی لینوکس (**cgroups**) استفاده می‌کند تا برای هر ماژول یک سقف مشخص تعیین کند.

| منبع | روش محدودیت | اگر از حد بگذرد |
|-------|-------------|------------------|
| **CPU** | سهم نسبی (مثلاً ۰.۵ هسته) | ماژول کندتر می‌شود، اما از کار نمی‌افتد |
| **RAM** | حداکثر حافظه (مثلاً ۵۱۲ مگابایت) | اگر از حد سخت عبور کند، **کرنل فرآیند را می‌کشد** (OOM Kill) |
| **Swap** | محدودیت swap (اختیاری) | کندی شدید، اما متوقف نمی‌شود |

هنگامی که ماژول به دلیل کمبود RAM کشته می‌شود:
- وضعیت ماژول در پنل به **«crashed»** تغییر می‌کند
- خطا در **سیستم لاگ متمرکز** ثبت می‌شود با عبارت `Out of Memory`
- ادمین می‌تواند بعد از افزایش محدودیت (در تنظیمات چرخ‌دنده)، ماژول را دوباره استارت کند.

تنظیمات منابع هر ماژول در همان فایل `site-layout.json` (در کنار اطلاعات پوشه‌بندی) ذخیره می‌شود. نیازی به فایل جداگانه نیست.

---

### لاگ متمرکز — عیب‌یابی سریع

وقتی ماژولی از کار می‌افتد یا خطایی می‌دهد، اولین چیزی که ادمین نیاز دارد دیدن لاگ است. ModuleHub یک **سیستم لاگ متمرکز** دارد:

- هر ماژول لاگ‌های خود را در `/var/log/modulehub/modules/<module-id>.log` ذخیره می‌کند
- در پنل چرخ‌دنده، دکمه **«📋 نمایش لاگ‌ها»** وجود دارد
- با یک کلیک، آخرین ۵۰ خط لاگ (شامل خطاهای اجرا، خروجی فرآیند، و رویدادهای مهم) نمایش داده می‌شود
- امکان دانلود فایل کامل لاگ برای بررسی عمیق‌تر وجود دارد

این لاگ‌ها شامل:
- خطاهای وابستگی (`ModuleNotFoundError`، `Cannot find module`)
- مشکل پورت (`EADDRINUSE`)
- خروجی خطای استاندارد (`stderr`)
- رویدادهای Start/Stop و آپدیت از GitHub

> ادمین بدون نیاز به دسترسی SSH به سرور، می‌تواند مشکل ماژول را تشخیص دهد.

---

### کش متمرکز پکیج‌ها — آپلود سبک‌تر و نصب سریع‌تر

یکی از مشکلات رایج در سیستم‌های ماژولار این است: هر بار که یک ماژول مشابه آپلود می‌کنید، مجبورید کل وابستگی‌های حجیم (مثل `node_modules` یا `venv`) را دوباره آپلود کنید. ModuleHub این مشکل را با **کش سطح سرور** حل می‌کند.

**چگونه کار می‌کند؟**

1. هنگام آپلود ZIP، سیستم فایل‌های وابستگی (`package.json`، `requirements.txt`، `composer.json`) را اسکن می‌کند.
2. یک **هش (hash)** از محتوای آن فایل‌ها می‌سازد.
3. اگر قبلاً همان هش روی سرور وجود داشته باشد (در `/var/cache/modulehub/pkg/<hash>/`)، از آن استفاده می‌کند و نیازی به نصب مجدد نیست.
4. در غیر این صورت، دستور نصب (مثل `npm install` یا `pip install -r requirements.txt`) را اجرا کرده و خروجی را در کش ذخیره می‌کند.

**مزایا برای ادمین:**
- حجم آپلود ماژول به شدت کاهش می‌یابد (فقط کد منبع، بدون وابستگی‌ها)
- نصب ماژول‌های تکراری روی سرور بسیار سریع‌تر می‌شود
- صرفه‌جویی در پهنای باند و زمان

> نکته: این کش برای همه ماژول‌های یک سرور مشترک است. با به‌روزرسانی وابستگی‌ها، هش تغییر می‌کند و کش جدید ساخته می‌شود؛ نسخه‌های قدیمی به‌طور خودکار پاک می‌شوند.

---

### تنظیمات سراسری — یک جا، بدون SSH

**چالش:** هر ماژول محدودیت خودش را دارد، اما سرور هم سقف کلی می‌خواهد — حجم ZIP، تعداد ماژول همزمان، نگه‌داری لاگ، و روی سرور دو خط اینترنت، نصب `npm` از کدام برود؟

**پاسخ:** صفحه **«تنظیمات و محدودیت‌ها»** (`/admin/settings`) — فقط ادمین اصلی. مثل تنظیمات مودم: یک بار تعیین می‌کنید، CMS رعایت می‌کند.

- ZIP خیلی بزرگ، → رد می‌شود (پیش‌فرض حدود ۲۰۰ مگ)
- پورت خالی در Add wizard → CMS خودش از محدوده ۴۱۰۰–۴۹۹۹ می‌دهد
- دو کارت شبکه → با radio انتخاب می‌کنید نصب پکیج از کدام خط برود؛ **مسیر دائمی Ubuntu عوض نمی‌شود** (Xray امن می‌ماند)

جزئیات اعداد، فیلدها و JSON: `design plan.md` §۱۰ و `system-settings.example.json`.

## کاربر عادی چه می‌بیند؟

- همه کارت‌ها را می‌بیند
- روی هر کارت کلیک می‌کند → وارد ماژول می‌شود
- **چرخ‌دنده ⚙ را نمی‌بیند**
- **کارت + را نمی‌بیند**
- هیچ پنل مدیریتی نمی‌بیند

دقیقاً همان چیزی که باید باشد: یک سایت تمیز با کارت‌های مرتب.

## تفاوت ماژول‌ها در پشت صحنه — خیلی ساده

| ماژول شما... | سیستم چه می‌کند |
|--------------|-----------------|
| فقط چند فایل HTML و CSS است | CMS مستقیم از پوشه ماژول سرو می‌کند — **بدون پورت** |
| برنامه React/Vue (SPA بیلدشده) | مثل Static + مسیرهای عمیق به `index.html` برمی‌گردند |
| بک‌اند دارد (Node, PHP, Python) | یک پروکسی مسیر را به پورت ماژول وصل می‌کند، و وابستگی‌ها را از طریق `npm install` یا `pip install -r requirements.txt` نصب می‌کند |
| از Docker استفاده می‌کند | داخل کانتینر با محدودیت CPU و RAM اجرا می‌شود |
| داده زنده (WebSocket) دارد | حتماً در Docker اجرا می‌شود تا اتصالات پایدار بماند |

**نکته:** شما **نمی‌گویید** ماژول شما چه نوعی است. فقط می‌گویید: «آیا Docker می‌خواهد؟ پورتش چند است؟ چه دسترسی‌هایی دارد؟» سیستم بقیه را خودش تشخیص می‌دهد.

## سازگاری با شرایط سرور شما (dual‑WAN و فیلترینگ)

روی سرور شما دو کارت شبکه وجود دارد: `ens4` (ADSL فیلترشده، metric پایین‌تر) و `enp63s0` (اینترنت آزاد، metric بالاتر). ModuleHub طوری طراحی شده که بدون نیاز به تغییر دستی مسیرها کار کند:

- **ماژول‌های معمولی (Static، SPA، Node بدون Docker)** از طریق `ens4` (پیش‌فرض) ارائه می‌شوند. این برای ترافیک داخلی و کاربران عادی کافی است.
- **ماژول‌هایی که نیاز به اینترنت آزاد دارند** (مانند `docker pull`، `git clone` از GitHub، `npm install` پکیج‌های خارجی) می‌توانند از یک سرویس کمکی استفاده کنند که به‌طور موقت **مسیر پیش‌فرض** را به `enp63s0` تغییر می‌دهد، کار را انجام می‌دهد و سپس مسیر را برمی‌گرداند.
- همچنین در تنظیمات **سراسری** (`/admin/settings`) رابط شبکه برای **نصب پکیج** انتخاب می‌شود (موقت — نه runtime هر ماژول).

> برای راحتی کار، ابزار خط فرمان `network-metric-toggler` در کنار CMS قرار می‌گیرد. ادمین با یک دستور می‌تواند مسیر آزاد را فعال یا غیرفعال کند.

---

## خلاصه نهایی

**ModuleHub CMS = صفحه اصلی + کارت‌های ماژولار**

- **برای ادمین:** کارت + → آپلود ZIP → پاسخ سه سوال → تعیین منابع (CPU/RAM) → ماژول جدید اضافه می‌شود. اگر خطایی رخ دهد، لاگ متمرکز و کش پکیج‌ها به عیب‌یابی و سرعت کمک می‌کنند.
- **برای کاربر:** فقط کارت‌ها را می‌بیند و کلیک می‌کند.
- **برای سایت:** کد اصلی هیچوقت تغییر نمی‌کند.

هر قابلیت جدید، یک ماژول جدید، یک کارت جدید. به همین سادگی.
