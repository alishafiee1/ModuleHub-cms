## پروپزال: سامانه مدولار مدیریت محتوای پویا با قابلیت ماژول‌های مستقل (Standalone) مبتنی بر داکر

> **به‌روزرسانی v2 (2026):** صفحه اصلی عمومی `/` با `site-layout.json`، ماژول‌های عادی **built-in** در `core/builtin-modules/`، آپلود ZIP فقط برای **standalone** (`index.html` + Docker). جزئیات: `openspec/changes/public-homepage-architecture-v2/`

### ۱. عنوان پروژه
**CMS پویای هسته‌محور با پشتیبانی از ماژول‌های ایزوله (Dockerized Standalone Modules)**  
نام پیشنهادی: `moduleHub-cms`

### ۲. چشم‌انداز و اهداف کلان
طراحی و پیاده‌سازی سامانه‌ای که به مدیر سایت اجازه دهد:
- صفحات و ماژول‌های مختلف (گالری، نمایشگر مارک‌دان، داشبوردهای لحظه‌ای، کنترل ربات و...) را به صورت **بلوک‌های مستقل** اضافه، حذف یا به‌روز کند.
- ماژول‌های پیچیده که نیاز به اجرای کد سمت سرور (پایتون، نود، و ...) و گوش دادن به پورت دارند، در **کانتینرهای داکر ایزوله** اجرا شوند.
- توسعه توسط افراد مختلف یا هوش مصنوعی ساده شود: هر ماژول یک پوشه با `index.html`، `manifest.json` و (در صورت نیاز) `docker-compose.yml` دارد.
- مدیریت منابع، دسترسی شبکه و وضعیت اجرای ماژول‌ها از طریق پنل گرافیکی بصری انجام شود.

### ۳. معماری پیشنهادی (خلاصه)

| لایه | مؤلفه | وظیفه |
| :--- | :--- | :--- |
| **هسته (Core)** | PHP/Node.js + JSON database | خواندن فهرست ماژول‌ها از `modules.json`، مدیریت درخواست‌ها، واسط با Docker API، پروکسی معکوس پویا |
| **ماژول‌های عادی** | پوشه شامل `index.html`، `manifest.json`، CSS، JS | صفحات ایستا یا سمت کاربر (مثل گالری تصاویر، نمایشگر وبلاگ) |
| **ماژول‌های مستقل** | پوشه شامل `manifest.json`، `docker-compose.yml` و کدهای سمت سرور | برنامه‌هایی با پردازش لحظه‌ای، WebSocket، API سفارشی که در کانتینر داکر اجرا می‌شوند |
| **زیرساخت اجرا** | Docker Engine + Docker Compose | راه‌اندازی، توقف، مانیتورینگ و ایزوله‌سازی ماژول‌ها |
| **پروکسی معکوس** | Traefik یا Nginx با قابلیت به‌روزرسانی پویا | مسیردهی درخواست‌های `/modules/<module-name>` به پورت داخلی ماژول |

### ۴. جریان کاری کامل از دید مدیر

1. **افزودن ماژول جدید**  
   مدیر از طریق پنل ادمین یک فایل ZIP شامل ساختار استاندارد آپلود می‌کند. هسته فایل‌ها را در `standalone-modules/<name>/` یا `static-modules/<name>/` اکسترکت می‌کند.

2. **اعتبارسنجی و درخواست دسترسی**  
   - فایل `manifest.json` خوانده می‌شود (نام، نوع، پورت‌های مورد نیاز، محدودیت حافظه و CPU، سطح دسترسی‌ها).  
   - اگر ماژول از نوع `standalone` باشد، هسته پیامی به مدیر نمایش می‌دهد:  
     *«این ماژول به پورت‌های ۳۰۰۰ و ۸۰۸۰ و دسترسی به شبکه خارج نیاز دارد. آیا اجازه می‌دهید؟»*  
   - مدیر می‌تواند دسترسی‌ها را بپذیرد، رد کند یا محدود کند (مثلاً فقط پورت ۳۰۰۰ باز شود).

3. **راه‌اندازی کانتینر**  
   - با تأیید مدیر، هسته در پوشه ماژول دستور `docker compose up -d` را صادر می‌کند.  
   - محدودیت‌های منابع (CPU، RAM) از طریق گزینه‌های `docker run` یا فیلد `deploy` در `compose` اعمال می‌شود.  
   - هسته از طریق Docker API پورت مپ شده به هاست را کشف می‌کند (مثلاً `localhost:32775`).

4. **اتصال به پروکسی معکوس**  
   - یک `entry` جدید در پیکربندی پروکسی اضافه می‌شود:  
     `site.com/modules/robot-dashboard/` → `http://localhost:32775`  
   - پروکسی به صورت خودکار (بدون ریستارت) توسط هسته به‌روز می‌شود.

5. **نمایش وضعیت در پنل اصلی**  
   - در صفحه اصلی سایت (پنل مدیر) یک بنر مربعی با عنوان، آیکون و **دایره سبز رنگ** در کنار ماژول‌های در حال اجرا نشان داده می‌شود.  
   - با هاور کردن روی دایره، مصرف CPU و RAM (دریافت از `docker stats`) به صورت tooltip نمایش می‌یابد.  
   - دکمه‌های **Stop / Start / Logs** در همان پنل تعبیه می‌شود.

6. **توقف و حذف ماژول**  
   - با فشردن دکمه Stop، هسته دستور `docker compose down` را اجرا می‌کند.  
   - با حذف ماژول از پنل، علاوه بر توقف کانتینر، پوشه مربوطه و ورودی پروکسی نیز حذف می‌شود.

### ۵. چالش‌ها و راهکارهای فنی مصوب

| چالش | راهکار |
| :--- | :--- |
| **ایمنی اجرای کد ناشناس در داکر** | استفاده از `manifest.json` اجباری با فیلدهای `cap_drop: ALL`، `read_only: true`، `network: internal` (مگر نیاز به اینترنت). همچنین ماژول‌ها در شبکه داکر ایزوله از هم قرار گیرند. |
| **تشخیص خودکار پورت و اتصال به پروکسی** | هسته بعد از `docker compose up`، پورت مپ شده را از `docker inspect` می‌خواند و API پروکسی (مثلاً Traefik) را به‌روز می‌کند. در صورت عدم وجود API، فایل کانفیگ Nginx بازنویسی و سرویس reload می‌شود. |
| **مدیریت منابع و مانیتورینگ بدون دغدغه** | استفاده مستقیم از `docker stats` برای هر ماژول؛ نیازی به ذخیره در JSON نیست. داده‌ها به صورت لحظه‌ای در پنل نمایش داده می‌شوند. |
| **هشدار فایروال برای دسترسی خارجی** | پس از راه‌اندازی، هسته پورت مپ شده را تشخیص داده و پیامی به مدیر نمایش می‌دهد: «ماژول روی پورت ۳۲۷۷۵ هاست در دسترس است. برای دسترسی از بیرون، این پورت را در فایروال باز کنید.» |
| **سازگاری با ماژول‌های عادی (غیر داکری)** | آن‌ها در پوشه `static-modules/` قرار می‌گیرند و هسته مستقیماً فایل `index.html` آن‌ها را سرو می‌کند. وضعیت آن‌ها همیشه «در حال اجرا» (دایره سبز) است چون نیازی به پروسس ندارند. |
| **واگذاری مدیریت هر ماژول به افراد خاص** | در `manifest.json` سطح دسترسی اضافه می‌شود: `"admin_role": "editor_robot"`. هسته هنگام ورود کاربر، فقط ماژول‌های متناسب با نقش او را نمایش می‌دهد و امکان استارت/استاپ می‌دهد. |

### ۶. ساختار پوشه‌ها و فایل‌های پیشنهادی

```
/site-root/
├── core/
│   ├── index.php                  # صفحه اصلی که modules.json را می‌خواند و بنرها را نمایش می‌دهد
│   ├── admin.php                  # پنل مدیریت ماژول‌ها
│   ├── modules.json               # فهرست ماژول‌ها با وضعیت و متادیتا
│   ├── docker-manager.php         # توابع ارتباط با Docker API
│   └── proxy-updater.php          # به‌روزرسانی پیکربندی پروکسی
├── static-modules/                # ماژول‌های عادی
│   └── sample-gallery/
│       ├── index.html
│       ├── manifest.json
│       └── style.css
├── standalone-modules/            # ماژول‌های مستقل داکری
│   └── robot-dashboard/
│       ├── docker-compose.yml
│       ├── manifest.json
│       ├── app.py                 # کد پایتون (در کانتینر کپی می‌شود)
│       └── requirements.txt
└── config/
    ├── nginx/                     # یا traefik
    └── firewall-rules.sh
```

**نمونه `manifest.json` برای ماژول مستقل:**
```json
{
  "name": "داشبورد کنترل ربات",
  "type": "standalone",
  "icon": "robot.png",
  "version": "1.0",
  "dockerComposeFile": "docker-compose.yml",
  "allowedPorts": [3000, 8080],
  "resources": {
    "memory": "512M",
    "cpu": "0.5"
  },
  "reverseProxy": {
    "prefix": "/modules/robot-dashboard/",
    "internalPort": 3000
  },
  "admin_role": "robot_engineer",
  "network_access": "outbound-limited"
}
```

**نمونه `docker-compose.yml` (ساده):**
```yaml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000"   # پورت بدون مشخص کردن هاست – داکر تصادفی مپ می‌کند
    deploy:
      resources:
        limits:
          memory: 512M
    read_only: true
    cap_drop:
      - ALL
```

### ۷. مزایای این معماری نسبت به راه‌حل‌های سنتی

- **توسعه موازی و ناهمگن:** هر ماژول می‌تواند با زبان و فریمورک جداگانه نوشته شود.
- **تسهیل استفاده از هوش مصنوعی:** به AI می‌توان گفت «یک ماژول گالری ویدیویی با پنل ادمین بساز و خروجی را به صورت پوشه با `manifest.json` تحویل بده» – بدون نیاز به درک کل پروژه.
- **مقیاس‌پذیری:** ماژول‌های مستقل را می‌توان روی سرورهای دیگر یا سرویس‌های ابری (مثل Kubernetes) نیز مستقر کرد.
- **امنیت ایزوله:** نقص امنیتی در یک ماژول (مثل SQL Injection در پایتون) به هسته یا سایر ماژول‌ها آسیب نمی‌زند.

### ۸. مراحل پیاده‌سازی پیشنهادی (فازبندی)

| فاز | مدت تخمینی | تحویل‌ها |
| :--- | :--- | :--- |
| **فاز ۱: هسته اولیه** | ۲ هفته | - نمایش بنرها از `modules.json` <br>- پنل آپلود فایل ZIP و اکسترکت <br>- اعتبارسنجی اولیه `manifest.json` |
| **فاز ۲: یکپارچه‌سازی داکر** | ۳ هفته | - اجرا/توقف ماژول‌های standalone با Docker Compose <br>- دریافت وضعیت و مصرف منابع از Docker API <br>- نمایش دایره سبز و ابزارک منابع |
| **فاز ۳: پروکسی معکوس پویا** | ۲ هفته | - به‌روزرسانی خودکار Nginx/Traefik <br>- مسیردهی `site.com/modules/name` به کانتینر |
| **فاز ۴: امنیت و نقش‌ها** | ۱ هفته | - پیاده‌سازی محدودیت‌های `manifest.json` (capabilities، حافظه، شبکه) <br>- سیستم نقش‌ها برای مدیریت ماژول توسط افراد خاص |
| **فاز ۵: تکمیل و مستندسازی** | ۱ هفته | - مستندات API برای توسعه‌دهندگان ماژول <br>- راهنمای نصب و اجرا روی سرور |


### ۹. نتیجه‌گیری و جمع‌بندی

پروپزال فوق یک **CMS مدولار با قابلیت افزودن ماژول‌های ایزوله داکری** را تشریح می‌کند که:
- به مدیر اجازه می‌دهد هر نوع صفحه (ایستا، دینامیک، لحظه‌ای، پردازشگر) را بدون تغییر در هسته اضافه کند.
- توسعه توسط تیم‌های پراکنده یا هوش مصنوعی را با ارائه یک قرارداد ساده (پوشه + `manifest.json`) ممکن می‌سازد.
- امنیت و مدیریت منابع را از طریق داکر و پروکسی معکوس تضمین می‌کند.

این راهکار در پروژه‌های مشابه (مانند **Coolify**، **CapRover** و حتی **WordPress با معماری پلاگین‌های Containerized**) تا حدی دیده می‌شود، اما نوآوری اصلی در سادگی مفهوم «هر ماژول = یک پوشه با داکر» و نمایش گرافیکی وضعیت برای مدیران غیرفنی است.
