# پروپزال: ModuleHub CMS

**سامانهٔ وب ماژولار برای نمایش نمونه‌کار، آموزش، رسانه و محتوای زنده — بدون دست‌زدن به کد هسته**

> جزئیات فنی: [design.md](design.md) · فهرست docs: [README.md](README.md) · راهنمای ماژول: [standalone-module-guide.md](standalone-module-guide.md)

---

## این پروژه برای چه کسی است؟

شما طراح وب، توسعه‌دهنده، یا صاحب کسب‌وکار هستید و می‌خواهید سایتی داشته باشید که:

- نمونه‌کارها، منابع آموزشی، ویدئوها، عکس‌ها و اخبار را به مخاطب نشان دهد
- محتوا در دسته‌های جدا و قابل فهم باشد
- برای هر قابلیت جدید لازم نباشد **کد سایت اصلی** را باز کنید
- هر بخش جدید — از هر نوعی (Static، Server-Dynamic، SPA، Live Widget، Standalone) — **جدا** اضافه شود و وابستگی‌ها و دسترسی‌ها به‌هم نریزد

**ModuleHub** همان «هستهٔ سایت» است: یک صفحهٔ اصلی با کارت‌ها، و پشت هر کارت یک ماژول مستقل.

---

## پوشه‌بندی مجازی — مثل Explorer ویندوز

> **مهم:** پوشه‌های سایت **مجازی** هستند و فقط در **JSON** (`site-layout.json`) تعریف می‌شوند — مثل درخت Explorer. روی دیسک برای هر پوشهٔ «آموزش» یا «نمونه‌کار» فولدر فیزیکی ساخته **نمی‌شود**.

- در **هر صفحه** (ریشه یا داخل یک پوشه) می‌توانید **پوشهٔ مجازی جدید** یا **ماژول** اضافه کنید.
- **پوشه** = فقط دسته‌بندی در JSON.
- **ماژول** = کارت + پوشهٔ واقعی مستقل روی سرور (`standalone-modules/<instance-id>/`)
- **Add** برای ادمین مثل **یک کارت در انتهای grid** است (آیکون +) — کاربر عادی نمی‌بیند.
- Add در **همان پوشهٔ فعلی** عمل می‌کند (breadcrumb مسیر را نشان می‌دهد — 🔜 فعلاً homepage فقط flat است).

```
/  →  📁 نمونه‌کار  →  🖼 گالری  |  📁 ۱۴۰۴  →  📄 md
                              ➕ Add (فقط ادمین)
```

---

## ایده در یک جمله

به‌جای اینکه هر بار کل پروژهٔ وب را تغییر دهید، ماژول جدید را در پوشهٔ خودش می‌سازید، از پنل admin اضافه می‌کنید، و هسته آن را روی سرور مستقر و مدیریت می‌کند.

---

## تجربهٔ مدیر سایت

### صفحهٔ فعلی و افزودن (Add)

در **انتهای کارت‌های همان صفحه** (ریشه یا داخل پوشهٔ مجازی)، ادمین کارت **Add** را می‌بیند:

| گزینه | نتیجه |
|--------|--------|
| **پوشهٔ جدید** | گره مجازی در JSON — بدون نصب |
| **آپلود ZIP** | ماژول شخصی + پوشه روی سرور |
| **کتابخانهٔ آماده** | گالری، مارک‌داون، … |

---

### مسیر ۱ — پروژهٔ شخصی (ZIP)

1. کد وب را در قالب ZIP آماده می‌کنید.
2. در محل دلخواه روی صفحه **Add** → آپلود ZIP.
3. هسته فایل‌ها را در `standalone-modules/<module-id>/` قرار می‌دهد و در JSON مسیریابی ثبت می‌کند.
4. **Docker برای تنظیمات** بالا می‌آید؛ **فرم تنظیمات** باز می‌شود (اگر manifest از قبل باشد، فیلدها خودکار پر می‌شوند). اگر تنظیمات ناقص بماند، کلیک روی کارت در `/admin` دوباره فرم را نشان می‌دهد.

| فیلد | توضیح |
|------|--------|
| پورت‌های موردنیاز | پورت‌هایی که برنامه داخل Docker می‌خواهد |
| دسترسی‌ها | شبکه، فایل، قابلیت‌های Linux |
| نام در مسیر سایت | مثلاً `/modules/my-app/` |
| سقف منابع | RAM و CPU مجاز |
| مخزن GitHub | برای به‌روزرسانی خودکار کد |
| فایل HTML اصلی | اولین صفحه‌ای که کاربر با کلیک می‌بیند |
| رمز پنل ماژول | اختیاری — برای مدیریت بدون login ادمین کل سایت |
| تصویر کارت | آیکون/تصویر روی homepage |

5. بعد از **تأیید**، ماژول در حالت **Running** ثبت می‌شود و API از مسیر proxy در دسترس است.

> **v2 فعلی:** upload + extract ✅ · Start/Stop دستی از admin ✅ · فرم settings mode 🔜

---

### مسیر ۲ — ماژول‌های آماده (کتابخانه)

ماژول‌هایی که **من Ali-shafiee** از قبل آماده کردم (عمدتاً Static یا Server-Dynamic، بدون Docker سنگین):

- **گالری تصاویر** — آپلود تکی یا گروهی عکس، حذف، مدیریت با رمز اختیاری
- **نمایشگر مارک‌داون** — لیست فایل‌های `.md`، پیش‌نمایش چند خط، صفحهٔ کامل با More

وقتی **Add** را می‌زنید، لیست از `core/catalog-modules/` (کتابخانهٔ قالب) خوانده می‌شود. با انتخاب یکی:

- سرور **کپی** از قالب را در `standalone-modules/<instance-id>/` می‌سازد — **هر instance کاملاً مستقل**
- فرم کوتاه: **نام کارت**، **تصویر کارت**، **رمز مدیر** (اختیاری)
- بقیه با سرور — بدون نیاز به ZIP و Docker برای شروع

هر نمونه (instance) می‌تواند مخزن Git جدا برای sync محتوا داشته باشد.

---

## مدیریت روزمره — چرخ‌دنده روی کارت

> **v2 فعلی:** Start/Stop/Logs روی homepage وقتی admin login کرده ✅ · دیالوگ ⚙ کامل 🔜

روی هر کارت، آیکون **⚙** وجود دارد (🔜). اگر برای ماژول رمز گذاشته باشید، همان را می‌پرسد؛ وگرنه (برای ادمین کل سایت) با session admin کافی است.

از این دیالوگ می‌توانید:

- **به‌روزرسانی از GitHub** — آخرین کد/محتوا را بکشید
- **ویرایش تنظیمات** — پورت، منابع، دسترسی‌ها، نام مسیر
- **Start / Stop / حذف** — سرویس و پوشه
- **آپلود ZIP جزئی یا کامل** — جایگزینی بخشی از فایل‌ها

> برای به‌روزرسانی از GitHub حتی لازم نیست حتماً به‌عنوان ادمین کل سایت login کنید — رمز همان ماژول کافی است.

## آنچه کاربر عادی می‌بیند

- **صفحهٔ جاری** — کارت **پوشه** (ورود به زیرمجموعه — 🔜) و **ماژول** (محتوا) ✅
- **breadcrumb** — مسیر پوشه‌های مجازی 🔜 (فعلاً فقط لیست flat در `/`)
- بدون **Add**، **⚙** و پنل admin ✅

## چرا ModuleHub؟

| مزیت | توضیح |
|------|--------|
| **جداسازی** | هر ماژول پوشه و lifecycle خودش |
| **بدون کدنویسی سایت** | افزودن از پنل، نه merge در repo اصلی |
| **انواع مختلف** | از صفحهٔ ساده تا اپ Dockerized |
| **امنیت** | ماژول‌های سنگین داخل Docker با محدودیت منابع |
| **آماده برای AI** | قرارداد ساده: پوشه + manifest + ZIP |

## نتیجه

ModuleHub CMS راهی است برای **شکل دادن به سایت بدون به‌هم‌ریختن پروژهٔ اصلی**: محتوا و قابلیت‌ها را ماژول‌به‌ماژول اضافه کنید، از پنل مدیریت کنید، و به مخاطب فقط تجربهٔ تمیز و دسته‌بندی‌شده نشان دهید.
