## ۶ — دیتابیس، JSON، و توضیح `config.json`

### قبلاً از دیتابیس استفاده نمی‌کرد؟

نه؛ **همیشه هر دو بوده‌اند**، ولی نقش‌شان فرق دارد:

```
تو در پنل 3x-ui تغییر می‌دهی
        ↓
   x-ui.db  (دیتابیس — جایی که پنل ذخیره می‌کند)
        ↓
   پنل از روی DB فایل config.json می‌سازد
        ↓
   Xray فقط config.json را می‌خواند و اجرا می‌کند
```

- **دیتابیس** (`/home/ash/3x-ui/db/x-ui.db`): جایی که inboundها، کاربران، پورت پنل، remark مثل `main-443` و… نگه‌داری می‌شود.
- **`config.json`**: خروجی نهایی برای موتور Xray — مثل «دستورالعمل اجرایی» که Xray فقط همین را می‌فهمد.

پس Xray مستقیم به دیتابیس وصل نیست؛ ولی **اگر JSON را دستی عوض کنی و بعد در پنل Save/Restart بزنی یا پنل دوباره config بسازد، احتمالاً تغییرت پاک می‌شود** چون پنل از DB دوباره می‌نویسد. راه درست: از پنل 3x-ui یا ابزارهایی که DB را به‌روز می‌کنند.

---

### بخش‌های مهم `config.json` — به زبان ساده

#### `log`
می‌گوید چقدر لاگ بنویسد. الان `warning` است — یعنی فقط خطاهای مهم.

#### `inbounds` — درهای ورود

**۱) پورت 62789 (api)** — فقط داخلی روی `127.0.0.1`  
برای اینکه خود پنل با Xray حرف بزند؛ کاربران به این وصل نمی‌شوند.

**۲) پورت 443 (اصلی VPN)**  
این همان دربی است که کاربران بهش وصل می‌شوند:

| فیلد | معنی |
|------|------|
| `protocol: vless` | نوع پروتکل VPN |
| `clients` + `id` (UUID) | کلید/رمز اتصال کاربر |
| `security: reality` | پنهان‌کاری — از بیرون شبیه HTTPS عادی به `www.haderbash.ir` دیده می‌شود |
| `serverNames: www.haderbash.ir` | نامی که کلاینت باید بزند (SNI) |
| `target: 127.0.0.1:8443` | اگر کسی Reality نزند، به nginx روی سرور خودت هدایت می‌شود (سایت نقاب) |
| `fallbacks` | مسیر جایگزین برای ترافیک غیر-Reality |

#### `outbounds` — درهای خروج

**`star` (اصلی)**  
`freedom` = مستقیم به اینترنت.  
**`sendThrough: 192.168.88.50`** = حیاتی: ترافیک کاربر از کارت `enp63s0` خارج می‌شود، نه از ADSL فیلتر.

**`khodam` (دوم)**  
یک سرور VLESS دیگر (`pv.callmesallad.store`) — مثل یک خط تلفن پشتیبان. در routing فعلی استفاده نمی‌شود مگر rule اضافه شود.

**`blocked`**  
سیاه‌چال — ترافیکی که اینجا بیفتد دور ریخته می‌شود. **الان ruleای بهش وصل نیست.**

#### `routing` — ترافیک کجا برود

الان فقط دو قانون:

1. api → api  
2. **همهٔ ترافیک VPN (in-443-tcp) → `star`**

یعنی همهٔ کاربران مستقیم از `enp63s0` به اینترنت می‌روند. هیچ بلاکی (تورنت، IP خصوصی) فعال نیست.

#### `dns`
وقتی کاربر داخل VPN سایتی را باز می‌کند، Xray خودش DNS می‌زند — با `8.8.8.8` و `1.1.1.1`. **جدا از DNS ویندوز/Ubuntu.**

#### `metrics` (پورت 11111)
آمار داخلی — فقط localhost.

## ۷ — بلاک `geoip:private` چطور ممکن است؟ الان می‌تواند؟

### ایده چیست؟

وقتی کسی با VPN وصل است، ترافیکش از سرور تو به اینترنت می‌رود. گاهی کاربر (یا بدافزار) سعی می‌کند به آدرس‌های **شبکهٔ داخلی** وصل شود — مثلاً `192.168.1.1` (مودم خانه‌اش)، `192.168.88.1` (روتر شبکهٔ تو)، یا `10.0.0.5`.

**`geoip:private`** یعنی: «اگر مقصد IP خصوصی/داخلی بود، اجازه نده.»

مثل اینکه به مهمان خانه بگویی: «می‌توانی از اینترنت استفاده کنی، ولی به اتاق‌های داخلی خانهٔ من در نرو.»

### الان می‌تواند؟

**خیر — در عمل فعال نیست.**

دلیل: outbound به نام `blocked` هست، ولی در `routing` هیچ قانونی نمی‌گوید «IP خصوصی → blocked». همه چیز مستقیم می‌رود به `star`.

### چطور فعالش کنی؟

از پنل 3x-ui (بخش Routing) یا با اضافه کردن rule در config، چیزی شبیه این:

```
اگر مقصد = geoip:private  →  برو به outbound «blocked»
اگر پروتکل = bittorrent   →  برو به outbound «blocked»
بقیهٔ ترافیک VPN         →  برو به «star»
```

بعد Save و Restart Xray از پنل. پنل دوباره `config.json` می‌سازد و Xray reload می‌شود.

**نکته:** بعد از فعال‌سازی، کاربر نمی‌تواند از طریق VPN به پرینتر یا NAS داخل LAN خودش (با IP خصوصی) برود — برای بعضی کاربرها مزیت است، برای بعضی مزاحمت.

## ۱۴ — DNS و 1.1.1.1 (دو لایه جدا)

بذار ساده بگم: سرورت **دو تا دفترچه تلفن** برای پیدا کردن آدرس سایت‌ها دارد؛ یکی برای خود Ubuntu، یکی برای Xray و کاربران VPN.

### دفترچهٔ اول — سیستم‌عامل (`/etc/resolv.conf`)

وقتی خود سرور می‌خواهد `docker pull` بزند، `apt update` کند، یا `git clone` کند، از این استفاده می‌کند. الان:

- `192.168.10.1` — مودم (سریع برای کارهای محلی)
- `178.22.122.100` — DNS شکن (برای سایت‌های فیلتر)
- `8.8.8.8` — پشتیبان

اینجا **1.1.1.1 نگذاشتی** و درست هم هست؛ در ایران Cloudflare (`1.1.1.1`) برای سیستم اغلب کند یا timeout می‌شود و docker/apt عصبانی می‌شوند.

### دفترچهٔ دوم — داخل Xray (`config.json`)

وقتی کاربر با VPN وصل است و مثلاً `youtube.com` را باز می‌کند، Ubuntu دخالت نمی‌کند؛ **خود Xray** می‌پرسد «آدرس YouTube چیه؟» و از `8.8.8.8` و `1.1.1.1` استفاده می‌کند.

این دو تا **مستقل** هستند. مثل اینکه تو برای خودت از یک اپ نقشه استفاده کنی، ولی مهمانت از اپ دیگر — هر کدام تنظیم خودش را دارد.

### اگر 1.1.1.1 داخل Xray کند شود چه؟

فقط **کاربران VPN** کند می‌شوند (باز کردن سایت‌ها)، نه خود سرور. اگر دیدی داخل VPN سایت‌ها دیر باز می‌شوند، در پنل 3x-ui بخش DNS را عوض کن و مثلاً `178.22.122.100` بگذار — نه در `resolv.conf` سیستم.

## ۱۵ — metric 1024، Outbound، و `ip route` (سه چیز جدا)

### مشکل Outbound چه بود؟

سرور دو خط اینترنت دارد: یکی ADSL فیلتر (`ens4`)، یکی آزادتر (`enp63s0`).  
بدون تنظیم خاص، ترافیک VPN هم از ADSL خارج می‌شد و کاربران فیلتر می‌ماندند.

**راه‌حل اصلی:** `sendThrough: 192.168.88.50` در outbound `star`.  
یعنی به Xray می‌گویی: «ترافیک کاربران را حتماً از IP کارت دوم بفرست.» این **مستقل از default route** است و مشکل فیلتر کاربران را حل کرد.

### `ip route add … metric 50` چیست؟

این برای **خود سرور** است، نه کاربران VPN. وقتی تو SSH می‌زنی و `docker pull` می‌خواهی، Ubuntu از مسیر پیش‌فرض (`ens4` / ADSL) می‌رود و شکست می‌خورد. با اضافه کردن موقت مسیر با metric 50 می‌گویی: «فعلاً از `enp63s0` برو.» بعد از کار با `del` برمی‌گردانی.

### metric 1024 چیست؟

وقتی `ip route show default` را می‌زنی، چهار خط می‌بینی. دو تا مهم‌اند (100 و 200). دو تای دیگر با **1024** هستند — DHCP خودش گذاشته، مثل «مسیر احتیاطی با اولویت خیلی پایین». عملاً استفاده نمی‌شوند چون 100 و 200 جلوترند.

**خلاصه:** metric 1024 ربطی به fix کردن Outbound نداشت. Outbound با `sendThrough` درست شد. metric 50 فقط برای کارهای خودت روی سرور (docker و…) است.

## ۱۶ — sudo broker یعنی چی؟ تو باید چه کار کنی؟

### مشکل

بعضی دستورها root می‌خواهند: `ip route`، `systemctl restart`، `ufw`، `apt`. هر بار `sudo` می‌زنی پسورد می‌خواهد — برای AI یا اسکریپت اذیت است.

### راه‌حل

یک برنامه کوچک (`sudo_broker.py`) پسورد را **یک بار** می‌گیرد و در حافظه نگه می‌دارد. بعد دستورات از `run_via_broker.py` بدون پسورد اجرا می‌شوند.

### تو چه کار کنی — عملی

**وقتی می‌خواهی از SSH کار privileged بکنی (یا به من/AI اجازه بدهی):**

**ترمinal ۱** — باز بماند:
```bash
cd ~/3x-ui
python3 sudo_broker.py
```
پسورد sudo را یک بار بزن. پیام `Broker ready` را ببین.

**ترمinal ۲** — دستورات عادی:
```bash
python3 ~/3x-ui/run_via_broker.py 'ip route show default'
python3 ~/3x-ui/run_via_broker.py 'systemctl restart modulehub-cms'
```

**وقتی تمام شد:** در ترمinal ۱ `Ctrl+C` بزن تا broker بسته شود (پسورد از RAM پاک می‌شود).

### چه موقع لازم است؟

| کار | broker لازم؟ |
|-----|----------------|
| `docker ps`، `curl`، `ip addr` | خیر |
| `ip route add`، restart سرویس، `ufw` | بله (یا sudo دستی) |
| وقتی AI از راه دور دستور privileged می‌زند | بله — همان‌طور که الان روشن کردی |

الان broker روشن است؛ همان کافی است تا کارهای admin بدون پسورد مکرر انجام شود.

**اصطلاحات ساده:**  
- **inbound/outbound:** ورودی و خروجی VPN  
- **routing rule:** قانون «این ترافیک کجا برود»  
- **blackhole/blocked:** سیاه‌چال — ترافیک دور ریخته می‌شود  
- **metric:** اولویت مسیر شبکه — عدد کمتر = مهم‌تر  
- **broker:** واسطه که یک بار پسورد root می‌گیرد

اگر بخواهی، می‌توانم rule بلاک `geoip:private` را از پنل قدم‌به‌قدم راه بیندازم، یا بخش ۶–۷ را به `server condition.md` اضافه کنم.
