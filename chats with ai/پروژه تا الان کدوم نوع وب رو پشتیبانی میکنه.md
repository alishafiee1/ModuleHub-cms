## نوع اول: سایت استاتیک (فقط فایل)

اینو ساده‌ترین حالت در نظر بگیر: یه فولدر داری که داخلش فقط این چیزهاست:

- `index.html`  
- `styles.css`  
- `main.js`  
- شاید چند تا عکس و فونت و...

اینجا هیچ برنامه‌ای روی سرور در حال اجرا نیست. [127.0.0](http://127.0.0.1:4001/help)
سرور فقط نقش یه «فایل‌سرور» رو بازی می‌کنه: هر کی اومد `/index.html` رو خواست، همین فایل رو براش می‌فرسته. [127.0.0](http://127.0.0.1:4001/help)

ویژگی‌ها:

- نه `npm install` لازم داری، نه `pip install`، نه دیتابیس، هیچی. [127.0.0](http://127.0.0.1:4001/help)
- هیچ پورتی به اسم «پروسه‌ی برنامه» نداری؛ همون وب‌سرور اصلی (مثلاً Nginx) فایل‌ها رو سرو می‌کنه. [127.0.0](http://127.0.0.1:4001/help)
- دیپلوی هم خیلی ساده‌ست: فقط فایل‌ها رو کپی می‌کنی توی یه مسیر روی سرور. [127.0.0](http://127.0.0.1:4001/help)

در ModuleHub-CMS:

- اگر یه ماژول رو طوری تعریف کنی که **نه docker داشته باشه، نه پورت مشخص**، سیستم فرض می‌کنه این یه ماژول استاتیکه. [127.0.0](http://127.0.0.1:4001/help)
- بعد خودش اون فولدر رو روی مسیر چیزی شبیه `/modules/<module-id>/` سرو می‌کنه. [127.0.0](http://127.0.0.1:4001/help)

این حالت برای لندینگ‌ها، داکیومنتیشن استاتیک و صفحه‌های ساده عالیه.

## نوع دوم: SPA بیلد شده (React/Vue و امثالهم)

این‌یکی از بیرون خیلی شبیه استاتیکه، ولی یه نکته مهم داره. [127.0.0](http://127.0.0.1:4001/help)
فرض کن با React یا Vue یه پروژه ساختی:

- توی لوکال با `npm run dev` میاریش بالا  
- برای دیپلوی، `npm run build` می‌زنی  
- آخرش یه پوشه مثل `dist/` یا `build/` می‌گیری

خروجی این بیلد در نهایت فقط یه سری فایل استاتیکه (HTML/CSS/JS)، ولی:

- روتر داخل خود مرورگر کار می‌کنه (History API)  
- یعنی وقتی کاربر میره `/dashboard` باید **همون `index.html` لود بشه** و JS خودش روتر رو هندل کنه. [127.0.0](http://127.0.0.1:4001/help)

دو تا نکته مهم برای این نوع پروژه:

1. Base path  
   چون ماژول توی ModuleHub-CMS زیر یه مسیر مثل `/modules/<id>/` سرو می‌شه، باید توی تنظیمات build بگی که base اینه. [127.0.0](http://127.0.0.1:4001/help)
   مثلا در Vite:

   ```ts
   export default defineConfig({
     base: '/modules/my-module/',
   })
   ```

2. Fallback به `index.html`  
   هر مسیری که کاربر میزنه (`/modules/my-module/dashboard`) باید در نهایت `index.html` رو برگردونه و روتر front-end ادامه کار رو انجام بده. [127.0.0](http://127.0.0.1:4001/help)

در ModuleHub-CMS:

- باز هم چون پورت و docker تعریف نشده، سیستم این رو مثل یه ماژول static/SPA می‌بینه، فقط توی تنظیمات nginx (یا معادلش) طوری route می‌کنه که هر چیز ناشناخته‌ای بره به `index.html`. [127.0.0](http://127.0.0.1:4001/help)

یعنی از دید سیستم، این هم تو دسته «بدون پروسه‌ی جداگانه» قرار می‌گیره، ولی با یه رفتار routing هوشمندتر. [127.0.0](http://127.0.0.1:4001/help)

## نوع سوم: Backend (برنامه‌ای که روی سرور اجرا می‌شود)

اینجا دیگه فقط فایل نیست، یه برنامه‌ی در حال اجرا داریم. [127.0.0](http://127.0.0.1:4001/help)
مثال‌ها:

- Node.js (Express, Nest, Koa, Fastify و غیره)  
- Python (Flask, Django, FastAPI)  
- PHP (با PHP-FPM) و...

اینجا سناریو اینه:

1. روی سرور یه سری dependency نصب می‌کنی (مثلاً `npm install`). [127.0.0](http://127.0.0.1:4001/help)
2. یه دستور برای start داری (مثلاً `npm start` یا `node server.js`). [127.0.0](http://127.0.0.1:4001/help)
3. برنامه روی یه پورت گوش می‌ده (مثلاً `localhost:4200`). [127.0.0](http://127.0.0.1:4001/help)
4. بعد Nginx یا reverse proxy هر درخواست رو پاس می‌ده به همین پورت. [127.0.0](http://127.0.0.1:4001/help)

در ModuleHub-CMS طبق کدی که نوشته‌ای:

- اگر ماژول یه `port` داشته باشه، سیستم می‌فهمه این یه **backend module** است. [127.0.0](http://127.0.0.1:4001/help)
- توی `detect-start-command` برای Node.js به شکل هوشمند دنبال این‌ها می‌گرده: [127.0.0](http://127.0.0.1:4001/help)
  - `package.json` → `scripts.start`  
  - اگر نبود، دنبال فایل‌هایی مثل `index.js` یا `server.js` می‌گرده  
  - از روی این‌ها یه دستور start می‌سازه (مثلاً `node index.js`) [127.0.0](http://127.0.0.1:4001/help)
- بعد با `systemd-run` یا معادلش، پروسه رو به عنوان سرویس پس‌زمینه روی سرور بالا می‌آره و ترافیک رو به اون پورت هدایت می‌کنه. [127.0.0](http://127.0.0.1:4001/help)

برای زبان‌های دیگه مثل Python/PHP:

- می‌تونی با فایل‌هایی مثل `requirements.txt` یا `app.py` تشخیص بدی چی هست، ولی تو کد فعلی‌ات تمرکز اصلی روی Node.js بوده و auto-detect فقط اون رو واقعاً پشتیبانی می‌کنه. [127.0.0](http://127.0.0.1:4001/help)
- برای بقیه زبان‌ها معمولاً نیاز می‌شه دستور start رو دستی در تنظیمات ماژول مشخص کنی. [127.0.0](http://127.0.0.1:4001/help)

## نوع چهارم: Docker (همه‌چیز توی کانتینر)

اینو می‌تونی full isolation در نظر بگیری. [127.0.0](http://127.0.0.1:4001/help)
به‌جای اینکه مستقیم روی سرور `npm install` یا `pip install` بزنی، همه‌چیز داخل یه image/docker-compose تعریف شده:

- `Dockerfile` یا `docker-compose.yml` داری  
- image خودش همه dependencyها رو توی خودش داره  
- سرور فقط `docker run` یا `docker-compose up` اجرا می‌کنه. [127.0.0](http://127.0.0.1:4001/help)

ویژگی‌ها:

- محیط اجرای هر ماژول جداست (ورژن Node، Python، حتی OS) [127.0.0](http://127.0.0.1:4001/help)
- راحت‌تر می‌تونی روی چند تا سرور مختلف deploy کنی، چون فقط docker لازم دارند. [127.0.0](http://127.0.0.1:4001/help)

در ModuleHub-CMS:

- اگر توی تنظیمات ماژول `docker: true` باشه، سیستم این ماژول رو از نوع **Docker-based** در نظر می‌گیره. [127.0.0](http://127.0.0.1:4001/help)
- بعد با توجه به کانفیگی که برای image و port mapping دادی، کانتینر رو بالا می‌آره و ترافیک رو به پورت host که به container map شده proxy می‌کنه. [127.0.0](http://127.0.0.1:4001/help)

## خلاصه‌ی رفتار ModuleHub-CMS از دید خودت

در کد `module-classifier` و `detect-start-command`، منطق تقریباً اینه: [127.0.0](http://127.0.0.1:4001/help)

1. اگر `docker: true` → این ماژول Docker است.  
2. اگر `docker` نبود ولی `port > 0` → این یه backend است (پروسس جدا + پورت).  
3. اگر نه docker بود نه port → این ماژول static/SPA است (فقط فایل).

از طرف دیگه، برای backendها:

- Node.js رو به صورت کامل auto-detect و auto-start می‌کنی (با `package.json`, `scripts.start`, `index.js` و غیره). [127.0.0](http://127.0.0.1:4001/help)
- برای زبان‌های دیگه مثل Python/PHP فعلاً باید یا config دستی بدی یا تو آینده منطق تشخیص رو توسعه بدی. [127.0.0](http://127.0.0.1:4001/help)

پس می‌تونیم بگیم ModuleHub-CMS تو نسخه فعلی‌اش این دسته‌ها رو به شکل خوبی ساپورت می‌کنه:

| دسته کلی | وضعیت پشتیبانی در CMS |
|---------|------------------------|
| Static سایت ساده | کامل، بدون نیاز به پورت و پروسه جدا  [127.0.0](http://127.0.0.1:4001/help) |
| SPA بیلد شده | کامل، با تنظیم base path و fallback  [127.0.0](http://127.0.0.1:4001/help) |
| Node.js backend | کامل، همراه با auto-detect و auto-start  [127.0.0](http://127.0.0.1:4001/help) |
| Backendهای دیگر (Python/PHP و…) | قابل انجام، ولی بیشتر با تنظیم دستی  [127.0.0](http://127.0.0.1:4001/help) |
| Docker-based modules | کامل، وقتی `docker: true` ست شده باشد  [127.0.0](http://127.0.0.1:4001/help) |

اگر بخوای در قدم بعدی می‌تونیم با هم بشینیم دقیقا طراحی کنیم برای Python (مثلاً FastAPI) منطق auto-detect و start چطور به همین سیستم اضافه بشه که UX پانل برای بقیه تیم هم یکنواخت بشه.  

دوست داری از بین این‌ها، برای کدوم نوع پروژه (مثلاً Node backend، SPA، یا Docker) یه مثال end-to-end دیپلوی روی ModuleHub-CMS قدم‌به‌قدم برات بنویسم؟
